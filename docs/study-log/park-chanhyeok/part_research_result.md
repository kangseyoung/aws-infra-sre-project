# 🛡️ AWS 인프라 프로젝트: Security / SRE 보조 파트 계획서

본 문서는 4인 초보자 팀이 2주간 진행하는 AWS 인프라 구축 프로젝트에서 **[Security / SRE 보조]** 담당자의 역할, 핵심 개념, 마일스톤 및 협업 포인트를 정의한 문서입니다. 단순 웹페이지 구동을 넘어 시스템의 보안성과 안정성을 확보하는 것을 목표로 합니다.

---

## 1. 전체 아키텍처에서의 역할

* **Security (보안):** 외부 위협으로부터 인프라를 보호하는 '방패' 역할을 합니다. 가상 방화벽(보안 그룹)을 타이트하게 제어하고, 안전한 암호화 통신 채널(HTTPS)을 형성합니다.
* **SRE 보조 (안정성):** 인프라의 가시성(Visibility)을 확보합니다. 시스템의 건강 상태를 감시(CloudWatch)하고, 장애 상황 발생 시 팀원들에게 즉시 전파하며, 서비스가 무너지지 않고 버틸 수 있는 최소한의 방어선(Auto Scaling 체계)을 메인 담당자와 함께 설계합니다.

---

## 2. 초보자가 꼭 알아야 할 핵심 개념

* **보안 그룹 (Security Group):** AWS의 인스턴스 단위 가상 방화벽입니다. Stateful 방식으로 동작하며, *"어떤 IP와 포트의 접근만 허용(Allow)할 것인가?"*를 정의합니다.
* **최소 권한의 원칙 (Principle of Least Privilege):** 사용자나 시스템에 작업 수행에 필요한 '최소한의 권한'만 부여하여 보안 사고의 피해를 최소화하는 원칙입니다. (예: Root 계정 사용 자제, 제한된 IAM 사용)
* **SSL/TLS (HTTPS):** 클라이언트와 서버 간의 데이터를 암호화하여 중간자 공격(Sniffing) 등을 방지하는 보안 프로토콜입니다.
* **가시성 (Visibility):** 시스템 내부의 리소스 상태(CPU, 메모리, 트래픽 흐름)와 로그를 모니터링하여, 문제가 발생했을 때 원인을 추적할 수 있도록 눈에 보이게 만드는 것입니다.

---

## 3. 2주 프로젝트 마일스톤 (Must vs Optional)

### 📌 Must (반드시 해야 할 필수 범위)
> **목표:** 외부 위협 차단 + 암호화 통신 + 최소한의 모니터링 및 알림 체계 구축

1. **네트워크 및 접근 보안 규정 정의**
   * **보안 그룹(SG) 최소화:** ALB는 `80/443`만 오픈, EC2는 오직 ALB로부터 오는 트래픽만 허용, SSH(`22`) 포트는 팀원들의 실제 공인 IP로만 접근 가능하도록 제한.
   * **IAM 계정 분리:** 권한 관리를 위해 Root 계정 공유를 금지하고, 팀원별로 필요한 권한만 가진 IAM 사용자 계정 발급.
2. **HTTPS 통신 구현**
   * AWS Certificate Manager(ACM)를 활용해 SSL/TLS 인증서를 발급받고, ALB에 리스너 규칙을 추가하여 `HTTPS (443)` 통신 완수.
3. **기본 가시성 확보 및 경보 (CloudWatch)**
   * EC2 CPU 사용량, ALB 리퀘스트 수를 관측할 수 있는 CloudWatch 대시보드 생성.
   * 특정 임계치(예: CPU 사용량 80% 이상) 도달 시 AWS SNS를 연동하여 팀원들의 **이메일 또는 메신저(슬랙/디스코드)**로 즉시 알림 전송.

### ✨ Optional (시간이 남으면 할 확장 범위)
> **목표:** 포트폴리오의 깊이를 더하고 신입 서버 엔지니어 수준의 경험 확보

1. **부하 테스트 및 Auto Scaling 구조 검증**
   * `Apache Bench(ab)` 또는 `Artillery` 등의 간단한 툴을 활용해 인프라에 인위적인 트래픽 부하 유발.
   * 임계치 도달 시 **EC2가 자동으로 Scale-out(증설)**되고, 부하가 낮아지면 다시 Scale-in되는 복구 흐름 확인 및 기록.
2. **애플리케이션 로그 가시성 확보**
   * EC2 내부에 CloudWatch Agent를 설치하여 Docker 기반 Nginx의 Access/Error 로그를 CloudWatch Logs로 중앙 수집 및 시각화.

## 4. 타 파트와의 협업 포인트

* **Compute + Network 파트:**
  * VPC 및 서브넷 설계 시, EC2 서버를 외부 노출이 없는 **Private 서브넷**에 배치하고 ALB만 Public 서브넷에 둘 것을 제안 및 조율.
  * 인스턴스 생성 시 적용할 보안 그룹(SG)의 인바운드/아웃바운드 규칙 사전 협의.
* **Terraform / IaC 파트:**
  * 정의한 보안 그룹 규칙, IAM 정책, CloudWatch Alarm 임계치 등의 설정 값을 코드로 자동화할 수 있도록 파라미터 형태로 정리하여 전달.

## 6. 다음 회의 공유용 요약 (3분 스크립트)

> "안녕하세요, 이번 프로젝트에서 Security 및 SRE 보조를 맡은 [이름]입니다. 
> 
> 저희 파트의 핵심 목표는 **'인프라의 문을 안전하게 잠그고, 서비스가 안정적으로 작동하는지 감시하는 것'**입니다. 2주 동안 제가 집중할 핵심 작업은 크게 세 가지입니다.
> 
> **첫째, 보안 그룹 타이트닝과 IAM 분리입니다.** ALB는 웹 트래픽만 받도록 제한하고, 메인 EC2 서버는 인프라 내부(Private 서브넷)에 숨겨 외부 공격에 직접 노출되지 않도록 제안하고 설정하겠습니다. 또한 팀원 전원에게 최소 권한의 IAM 계정을 부여하겠습니다.
> **둘째, HTTPS 암호화 적용입니다.** ACM 인증서를 발급받아 ALB에 연동함으로써 전 구간 보안 통신을 완성하겠습니다.
> **셋째, CloudWatch 기반의 모니터링 구축입니다.** CPU나 트래픽에 경보가 발생하면 즉시 우리 팀원들의 이메일이나 메신저로 알림이 오도록 연동하겠습니다.
> 
> 인프라와 네트워크를 담당하시는 분과는 초반 VPC 및 서브넷 설계 단계부터 포트와 배치 규칙을 맞추어야 하므로, 오늘 회의가 끝난 후 바로 싱크를 맞췄으면 좋겠습니다.
> 
> 만약 이 필수 작업들이 빠르게 마무리된다면, 포트폴리오 퀄리티를 한 단계 높이기 위해 **'부하 테스트를 통한 Auto Scaling 자동화 체계'**까지 구현해 보는 것을 확장 목표로 가져가겠습니다. 감사합니다."
