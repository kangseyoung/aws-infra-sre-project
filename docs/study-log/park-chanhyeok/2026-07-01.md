## 2. 이번 학습의 전체 주제

<details>
<summary>내용 보기</summary>

이번 학습의 중심은 **AWS에서 서비스를 안정적으로 띄우기 위한 기본 인프라 구조**였습니다.

특히 아래 흐름을 중심으로 정리했습니다.

```text
사용자
↓
Internet
↓
Internet Gateway
↓
ALB
↓
Private Subnet의 EC2 서버
↓
DB / Redis / S3 같은 데이터 저장소
```

핵심적으로 이해한 내용은 다음과 같습니다.

```text
Region / AZ = 물리적 위치 단위
VPC / Subnet = 가상 네트워크 배치 단위
ALB = 외부 요청을 받아 살아 있는 서버로 분산
EC2 = 백엔드 애플리케이션 로직 실행
DB = 실제 유저 데이터 저장
Docker / ASG = 같은 서버를 여러 AZ에 배포하고 유지
```

</details>

## 3. 네트워크 구조 정리

<details>
<summary>내용 보기</summary>

### 3-1. Region, AZ, VPC, Subnet

AWS에는 여러 **Region**이 있고, 각 Region 안에는 여러 **AZ**가 있습니다.

```text
Region = 지리적 지역
AZ = Region 안의 물리적으로 분리된 데이터센터 묶음
```

반면 **VPC**는 물리적인 장비가 아니라, 특정 Region 안에서 정의되는 **가상 사설 네트워크**입니다.

```text
VPC = Region 단위의 가상 네트워크
Subnet = VPC 일부를 특정 AZ에 배치한 것
```

중요한 규칙:

```text
하나의 Subnet은 반드시 하나의 AZ에만 속한다.
```

예시:

```text
서울 Region
└─ VPC: 10.0.0.0/16

   AZ-1
   ├─ Public Subnet A
   └─ Private Subnet A

   AZ-2
   ├─ Public Subnet B
   └─ Private Subnet B
```

Subnet을 하나의 AZ에만 묶는 이유는 **물리적 위치와 장애 범위를 명확하게 나누기 위해서**입니다.

```text
AZ-1 장애 발생
↓
Private Subnet A는 영향 받음
Private Subnet B는 영향 없음
```

---

### 3-2. Public Subnet과 Private Subnet

Public / Private Subnet의 차이는 물리적 위치가 아니라 **인터넷과 직접 연결되느냐**입니다.

```text
Public Subnet
= 인터넷과 만나는 입구 구역

Private Subnet
= 백엔드 서버를 숨겨두는 내부 구역
```

보통 배치는 다음과 같습니다.

```text
Public Subnet
- ALB
- NAT Gateway
- Bastion Host

Private Subnet
- 백엔드 EC2
- 내부 서비스
- DB 계층
```

외부 사용자는 Private Subnet의 EC2에 직접 접근하지 않습니다.

```text
사용자
↓
ALB
↓
Private Subnet의 EC2
```

---

### 3-3. Internet Gateway와 ALB 차이

Internet Gateway와 ALB는 둘 다 외부 통신에 관여하지만 역할이 다릅니다.

```text
Internet Gateway, IGW
= VPC와 인터넷을 연결하는 가장 바깥 출입문
= L3 수준의 인터넷 연결 통로

ALB
= Public Subnet에 걸쳐 배치되는 스마트 안내원
= HTTP/HTTPS 요청을 읽고 적절한 EC2로 분산
```

즉:

```text
IGW = 인터넷 길을 열어주는 장치
ALB = 웹 요청을 받아 서버로 나눠주는 장치
```

</details>

## 4. ALB와 Bastion Host 역할 정리

<details>
<summary>내용 보기</summary>

### 4-1. ALB

ALB는 **Application Load Balancer**입니다.

핵심 역할:

```text
1. 외부 HTTP/HTTPS 요청을 받는다.
2. 뒤에 있는 여러 EC2로 트래픽을 나눠준다.
3. Health Check로 죽은 EC2를 제외한다.
4. Private Subnet의 EC2를 직접 노출하지 않게 해준다.
```

ALB는 하나의 Public Subnet에만 있는 것이 아니라, 여러 AZ의 Public Subnet에 걸쳐 동작합니다.

```text
AZ-1 Public Subnet A ─┐
                      ├─ 하나의 ALB처럼 동작
AZ-2 Public Subnet B ─┘
```

ALB의 Health Check:

```text
ALB → EC2 A 살아있냐?
ALB → EC2 B 살아있냐?

EC2 A 장애
↓
ALB가 EC2 A로 트래픽 보내지 않음
↓
EC2 B로 요청 전달
```

---

### 4-2. Bastion Host

Bastion Host는 개발자나 운영자가 **Private Subnet의 EC2에 관리 목적으로 접속하기 위한 우회용 서버**입니다.

```text
사용자 웹 요청
→ ALB, 80/443 포트

개발자 서버 관리
→ Bastion Host, 22번 SSH 포트
```

역할 차이:

```text
ALB
= 일반 사용자 웹 트래픽용 통로

Bastion Host
= 개발자/운영자 SSH 관리용 통로
```

Bastion Host의 22번 포트는 해커가 스캔할 수 있으므로, 보안 그룹에서 팀원의 공인 IP만 허용해야 합니다.

```text
SSH 22번 포트
Source = 우리 팀 공인 IP만 허용
```

</details>

## 5. EC2, Linux, Docker 정리

<details>
<summary>내용 보기</summary>

### 5-1. EC2

EC2는 AWS가 빌려주는 가상 컴퓨터입니다.

설치하는 프로그램에 따라 용도가 달라집니다.

```text
Bastion용 EC2
= SSH 접속 중계 서버

웹 서버용 EC2
= Node.js / Spring Boot 같은 백엔드 실행

DB용 EC2
= 직접 MySQL 같은 DBMS 설치 가능
```

하지만 실무에서는 DB를 EC2에 직접 설치하기보다 RDS 같은 관리형 서비스를 쓰는 경우가 많습니다.

---

### 5-2. Linux와 터미널

서버 시장에서 Linux가 많이 쓰이는 이유:

```text
1. 오픈소스라 비용 부담이 적음
2. GUI 없이 터미널만으로 가볍게 운영 가능
3. 자동화와 원격 제어에 강함
```

AWS 서버는 보통 마우스/화면이 아니라 SSH와 터미널 명령어로 제어합니다.

```text
ssh
ls
cd
mkdir
vim
systemctl
docker
```

---

### 5-3. Docker

Docker는 애플리케이션과 실행 환경을 컨테이너 이미지로 묶는 기술입니다.

핵심 역할:

```text
내 컴퓨터에서 되던 실행 환경을
서버에서도 똑같이 재현한다.
```

비유:

```text
AWS EC2 = 빈 방
Docker Image = 가구와 인테리어까지 정해진 박스
Container = 그 박스를 실제로 실행한 상태
```

Docker를 쓰면 여러 EC2에 같은 백엔드 앱을 쉽게 배포할 수 있습니다.

```text
AZ-1 EC2 → backend-app:v1.0.0 실행
AZ-2 EC2 → backend-app:v1.0.0 실행
```

</details>

## 6. 서버 고가용성 정리

<details>
<summary>내용 보기</summary>

### 6-1. 고가용성이란?

고가용성, HA, High Availability는 **장애가 나도 서비스가 최대한 계속 사용 가능하게 만드는 설계**입니다.

```text
고가용성 = 하나가 죽어도 다른 쪽이 이어받는 구조
```

중요한 개념:

```text
SPOF = Single Point of Failure
= 이것 하나 죽으면 전체 서비스가 죽는 지점
```

고가용성 설계는 SPOF를 줄이는 방향입니다.

```text
EC2 1대만 두지 않기
DB 1대만 두지 않기
ALB를 한 AZ에만 두지 않기
```

---

### 6-2. 서버 로직은 “동일 배포”

AZ-1의 EC2가 죽었을 때 AZ-2의 EC2가 대신 처리하려면, 두 EC2가 같은 백엔드 로직을 실행해야 합니다.

```text
AZ-1 EC2 A → backend-app:v1.0.0
AZ-2 EC2 B → backend-app:v1.0.0
```

중요한 점:

```text
서버 로직은 실시간 동기화가 아니라 동일 배포에 가깝다.
```

즉, EC2끼리 서로 코드를 계속 복사하는 것이 아닙니다.

```text
GitHub 코드
↓
Docker Image 생성
↓
이미지 저장소에 업로드
↓
AZ-1 EC2와 AZ-2 EC2가 같은 이미지 실행
```

---

### 6-3. Auto Scaling Group, ASG

ASG는 EC2를 자동으로 생성하고 유지하는 기능입니다.

예시 설정:

```text
최소 EC2 수: 2
대상 Subnet: Private Subnet A, Private Subnet B
Launch Template: 동일한 서버 실행 환경
```

결과:

```text
Private Subnet A에 EC2 1대
Private Subnet B에 EC2 1대
```

EC2 하나가 죽으면 ASG가 새 EC2를 띄워서 원하는 개수를 유지합니다.

```text
ASG = 같은 서버를 여러 AZ에 자동 배치하고 유지하는 장치
```

</details>

## 7. 데이터 저장소 계층 정리

<details>
<summary>내용 보기</summary>

### 7-1. 서버와 DB의 차이

EC2 서버는 데이터를 영구 저장하는 곳이 아니라, 백엔드 로직을 실행하는 곳입니다.

```text
EC2가 가진 것
- 애플리케이션 코드
- API 처리 로직
- 인증/인가 로직
- DB에 요청하는 코드
- 설정값
```

EC2가 직접 들고 있으면 안 되는 것:

```text
회원 정보
주문 내역
게시글
댓글
결제 정보
이미지 파일
```

실제 데이터는 별도 저장소에 둡니다.

```text
EC2 = 로직 실행
데이터 저장소 = 실제 데이터 보관
```

---

### 7-2. RDS, Redis, Aurora, S3

헷갈렸던 데이터 관련 서비스 정리:

```text
RDS
= AWS가 MySQL/PostgreSQL 같은 관계형 DB를 대신 운영해주는 서비스

Redis
= 빠른 임시 저장소/캐시

Aurora
= AWS가 만든 고성능 관계형 DB 엔진

S3
= 이미지/파일 저장소
```

비유:

```text
RDS = 공식 장부
Redis = 책상 위 임시 메모지
Aurora = AWS식 고성능 장부 시스템
S3 = 파일 창고
```

배달앱 예시:

```text
회원 정보, 주문 내역, 결제 내역 → RDS / Aurora
로그인 세션, 인기 가게 목록 캐시 → Redis
메뉴 사진, 리뷰 사진, 프로필 사진 → S3
```

</details>

## 8. RDS Multi-AZ와 Aurora 정리

<details>
<summary>내용 보기</summary>

### 8-1. 일반 RDS Multi-AZ

일반 RDS Multi-AZ는 다음처럼 이해할 수 있습니다.

```text
AZ-1
└─ Primary DB

AZ-2
└─ Standby DB
```

평소:

```text
EC2 → Primary DB
Primary DB 변경사항 → Standby DB에 복제
```

장애 발생:

```text
Primary DB 장애
↓
Standby DB가 새 Primary로 승격
```

한 줄 정리:

```text
RDS Multi-AZ = 일하는 DB 한 세트 + 대기 DB 한 세트
```

---

### 8-2. Aurora

Aurora는 DB 엔진과 저장소를 분리해서 생각해야 합니다.

```text
Writer Instance
= SQL 처리 엔진, 쓰기 담당

Reader Instance
= 읽기 담당

Cluster Volume
= 여러 AZ에 복제된 논리적 공유 저장소
```

그림:

```text
Writer / Reader Instance
          ↓
   Cluster Volume
          ↓
AZ-1 Storage Nodes
AZ-2 Storage Nodes
AZ-3 Storage Nodes
```

중요한 차이:

```text
일반 RDS Multi-AZ
= DB 인스턴스와 저장소가 한 세트로 묶여 있고,
  그 세트를 다른 AZ에 하나 더 둔다.

Aurora
= DB 엔진과 저장소를 분리하고,
  저장소 계층 자체를 여러 AZ에 복제한다.
```

---

### 8-3. Aurora에서 Primary는 무엇인가?

Aurora에서 Primary는 Storage가 아니라 **Writer Instance**입니다.

```text
쓰기 대장 = Writer Instance
저장소 대장 = 특정 AZ Storage가 아니라 Cluster Volume 전체
```

잘못된 이해:

```text
AZ-1 Storage = Primary Storage
AZ-2 Storage = Replica Storage
AZ-3 Storage = Replica Storage
```

올바른 이해:

```text
Cluster Volume 전체가 하나의 논리적 저장소처럼 동작한다.
특정 AZ의 Storage 하나를 Primary로 정하지 않는다.
```

---

### 8-4. Cluster Volume과 Storage Node

Cluster Volume은 Storage Node 위에 따로 있는 물리 저장소가 아닙니다.

```text
Cluster Volume
= 여러 Storage Node들을 묶어서 하나의 저장소처럼 보이게 만든 논리적/가상 볼륨
```

즉:

```text
Cluster Volume에 데이터가 있다
= Cluster Volume을 구성하는 여러 Storage Node들에 데이터가 저장되어 있다
```

Storage Node는 실제 데이터 page 복사본이 저장되는 곳입니다.

```text
Storage Node = 실제 데이터 page 복사본 저장소
Cluster Volume = 그 Storage Node 묶음을 하나처럼 보여주는 논리적 이름
```

---

### 8-5. Aurora에서 글 하나를 추가하면

게시판 글 작성 예시:

```sql
INSERT INTO posts (title, content, user_id)
VALUES ('안녕하세요', '첫 글입니다', 7);
```

흐름:

```text
사용자
↓
ALB
↓
EC2 백엔드 서버
↓
Aurora Writer Instance
↓
Cluster Volume
↓
여러 AZ의 Storage Node에 복제
```

단계:

```text
1. 사용자가 글 작성
2. EC2가 Writer Instance에 INSERT 요청
3. Writer Instance가 SQL 처리
4. 변경사항을 Cluster Volume에 기록
5. Cluster Volume을 구성하는 Storage Layer가 여러 AZ에 복제
6. 저장 성공 응답
```

---

### 8-6. Aurora에서 데이터를 읽으면

게시글 목록 조회 예시:

```sql
SELECT title, content
FROM posts
WHERE user_id = 7;
```

흐름:

```text
사용자
↓
ALB
↓
EC2 백엔드 서버
↓
Aurora Reader Instance 또는 Writer Instance
↓
Cluster Volume
↓
결과 반환
```

중요한 점:

```text
EC2가 Storage Node를 직접 고르지 않는다.
EC2는 DB Instance에 SQL 요청을 보낸다.
DB Instance가 Cluster Volume에서 필요한 데이터를 읽는다.
```

내부적으로 실제 데이터는 Cluster Volume을 구성하는 Storage Node들에 있지만, 사용자는 그것을 직접 선택하지 않습니다.

---

### 8-7. “조립한다”의 의미

Reader Instance가 결과를 “조립한다”는 말은 Storage Node마다 다른 내용을 들고 있어서 퍼즐 맞추듯 조립한다는 뜻이 아닙니다.

잘못된 이해:

```text
AZ-1 Storage에서 제목
AZ-2 Storage에서 내용
AZ-3 Storage에서 작성자
→ Reader가 조립
```

올바른 이해:

```text
Reader Instance가 data page를 읽고,
그 안에서 필요한 row와 column을 뽑아서
SQL 결과표로 만들어준다.
```

DB는 보통 row 하나만 직접 읽는 것이 아니라 page 단위로 읽습니다.

```text
Data Page 100번
├─ posts row id=1
├─ posts row id=2
└─ posts row id=3
```

그리고 그 page 안에서 필요한 데이터를 골라 SQL 결과로 만듭니다.

</details>

## 9. 새로 알게 된 점

<details>
<summary>내용 보기</summary>

이번 학습에서 새로 정리한 핵심은 다음과 같습니다.

```text
1. VPC는 물리 장비가 아니라 Region 단위의 가상 네트워크다.
2. Subnet은 반드시 하나의 AZ에만 속한다.
3. ALB는 여러 AZ의 Public Subnet에 걸쳐 동작한다.
4. EC2는 데이터를 저장하는 곳이 아니라 로직을 실행하는 곳이다.
5. 서버 코드는 실시간 동기화가 아니라 동일 배포한다.
6. DB 데이터는 운영 중 계속 바뀌므로 복제 구조가 필요하다.
7. RDS Multi-AZ는 Primary + Standby 구조다.
8. Aurora는 Writer/Reader Instance + Cluster Volume 구조다.
9. Aurora에서 Primary는 Storage가 아니라 Writer Instance다.
10. Cluster Volume은 여러 Storage Node 묶음을 하나처럼 보여주는 논리적 볼륨이다.
```

</details>

## 11. Must / Optional 마일스톤 후보

<details>
<summary>내용 보기</summary>

### 📌 Must 후보

1. **[VPC / Subnet / ALB / EC2 기본 구조 설계]**
   * 하나의 VPC 안에 2개 이상의 AZ를 사용하고, 각 AZ에 Public Subnet과 Private Subnet을 배치한다.
   * ALB는 Public Subnet에 걸치고, 백엔드 EC2는 Private Subnet에 배치한다.

2. **[보안 그룹 체이닝 및 IP 화이트리스트 적용]**
   * ALB는 외부 HTTP/HTTPS 요청을 받는다.
   * Private EC2는 ALB Security Group에서 오는 트래픽만 허용한다.
   * Bastion Host의 SSH 22번 포트는 팀원 공인 IP만 허용한다.

3. **[서버 동일 배포 구조 검토]**
   * Docker Image 또는 AMI를 사용해 AZ-1과 AZ-2의 EC2가 같은 서버 로직을 실행하도록 한다.
   * ASG를 통해 최소 2대의 EC2가 서로 다른 AZ에 유지되도록 한다.

### ✨ Optional 후보

1. **[단일 AZ 장애 대응 구조 검토]**
   * AZ-1 장애 시 ALB가 AZ-2 EC2로 트래픽을 넘길 수 있는 구조를 사전 검토한다.

2. **[RDS Multi-AZ 또는 Aurora 도입 검토]**
   * 실제 데이터 저장소 계층에서 Multi-AZ 구성을 적용할 수 있는지 검토한다.
   * 일반 RDS Multi-AZ와 Aurora 중 프로젝트 규모에 맞는 선택지를 비교한다.

3. **[비 웹 프로토콜 대응 구조 검토]**
   * HTTP/HTTPS 외 TCP 기반 서비스가 필요할 경우 NLB 도입 가능성을 검토한다.

</details>

<summary>🔍 Q1. ALB는 어디에 위치해있고, 어떤 역할을 한다는거야?</summary>

> **Q1. ALB는 어디에 위치해있고, 어떤 역할을 한다는거야?**
>
> **A1.** ALB(Application Load Balancer)는 외부 인터넷과 소통할 수 있도록 **퍼블릭 서브넷(Public Subnet)에 걸쳐서 위치**하며, 핵심 역할은 크게 3가지입니다.
> 1. **트래픽 분산:** 들어온 웹 요청을 뒤에 있는 여러 대의 쌍둥이 EC2 인스턴스에 공평하게 나누어 줍니다.
> 2. **보안 가림막:** 메인 서버들을 외부와 단절된 프라이빗 서브넷에 숨겨두고 ALB가 통신을 중계함으로써, 외부 공격으로부터 서버를 보호합니다.
> 3. **L7 스마트 라우팅:** HTTP/HTTPS 프로토콜을 이해하여 URL 주소나 헤더를 읽고 목적지별로 트래픽을 분기하거나 HTTPS 리디렉션을 처리합니다.
</details>

<details>
<summary>🔍 Q2. 인터넷 게이트웨이가 약간 ALB랑 헷갈리기 좋다고 들었었는데, ALB는 퍼블릭 서브넷에 걸쳐져 있는거고 인터넷 게이트웨이는 진짜 인터넷과 VPC 가장 마지막에 있는 약간 인터페이스같은 느낌인가?</summary>

> **Q2. 인터넷 게이트웨이가 약간 ALB랑 헷갈리기 좋다고 들었었는데, ALB는 퍼블릭 서브넷에 걸쳐져 있는거고 인터넷 게이트웨이는 진짜 인터넷과 VPC 가장 마지막에 있는 약간 인터페이스같은 느낌인가?**
>
> **A2.** 정확합니다. 두 서비스 모두 외부 통신에 관여하지만 역할과 레이어가 완전히 다릅니다.
> * **인터넷 게이트웨이 (IGW):** VPC의 가장 바깥 테두리에 있는 **'총출입문(인터페이스)'**입니다. 인프라 수준(L3)에서 인터넷 길을 뚫어주는 역할만 하며, 트래픽을 분산하는 스마트한 기능은 없습니다.
> * **ALB:** 대문을 열고 들어왔을 때 퍼블릭 서브넷에서 손님을 맞이하는 **'스마트 안내원(문지기)'**입니다. 애플리케이션 수준(L7)에서 패킷 내부(HTTP 텍스트)를 정독하고 배분합니다.
</details>

<details>
<summary>🔍 Q3. 한대에 요청이 무자비하게 쏟아질수도 있는건데, 그걸 다른 컴퓨터한테 보내도 돼? 그냥 애초에 해커가 걱정이면 private subnet에 납두면 되잖아.</summary>

> **Q3. 한대에 요청이 무자비하게 쏟아질수도 있는건데, 그걸 다른 컴퓨터한테 보내도 돼? 그냥 애초에 해커가 걱정이면 private subnet에 납두면 되잖아.**
>
> **A3.** > * **요청 분산:** ALB 뒤에 묶인 EC2들은 **동일한 소스코드가 돌아가고 있는 복제인간(쌍둥이) 컴퓨터들**이기 때문에, 사용자는 뒤에서 컴퓨터가 바뀌어도 똑같은 결과를 받으므로 전혀 문제없습니다.
> * **프라이빗 격리:** 메인 EC2들을 프라이빗 서브넷에만 가둬두면 외부 인터넷에서 직접 찾아올 수 있는 길이 끊겨 일반 고객도 서비스를 쓸 수 없습니다. 그래서 외부 손님이 합법적으로 노크할 수 있는 창구(Public 영역)로 ALB를 앞에 세워두는 것입니다.
</details>

<summary>🔍 Q4. L7 layer인데, application layer라는건, 이건 진짜 소프트웨어, 그니까 프로그램이라는거아니야? 웹 정보만 ALB가 통과시켜준다고? 그러면 SMTP같은애가 오면 내부로 통과 안시켜주는거야?</summary>

> **Q4. L7 layer인데, application layer라는건, 이건 진짜 소프트웨어, 그니까 프로그램이라는거아니야? 웹 정보만 ALB가 통과시켜준다고? 그러면 SMTP같은애가 오면 내부로 통과 안시켜주는거야?**
>
> **A4.** > * **L7의 본질:** 맞습니다. 내부적으로 Nginx나 Envoy 같은 고성능 프록시 소프트웨어 엔진이 돌아가며, 편지봉투(HTTP 텍스트)를 직접 뜯어서 내용을 읽고 판단합니다.
> * **SMTP 차단:** ALB는 철저하게 웹(HTTP/HTTPS)만 아는 전문가입니다. 메일 프로토콜인 SMTP 등이 들어오면 해석 자체가 불가능해 **그 자리에서 문전박대(드롭)**해 버립니다. 이 점이 오히려 웹 서버의 보안성을 극대화해 줍니다. (SMTP 등을 쓰려면 L4 수준의 NLB를 써야 합니다.)
</details>

<summary>🔍 Q5. 외부에서 들어온 요청을 private과 연결시켜주는 통로가 배스천 호스트(Bastion Host)인 줄 알았는데, ALB랑 역할이 어떻게 다른 거야? 그리고 우리 팀 IP로 22번 포트를 막는 거랑 무슨 연관이 있지?</summary>

> **Q5. 외부에서 들어온 요청을 private과 연결시켜주는 통로가 배스천 호스트(Bastion Host)인 줄 알았는데, ALB랑 역할이 어떻게 다른 거야? 그리고 우리 팀 IP로 22번 포트를 막는 거랑 무슨 연관이 있지?**
>
> **A5.** 외부 통신의 목적에 따라 들어오는 문과 통로가 완전히 분리됩니다.
> * **손님용 통로 (웹 트래픽):** 사용자 ➡️ ALB (`80/443` 포트) ➡️ 프라이빗 메인 EC2
> * **개발자용 통로 (서버 관리):** 개발팀 ➡️ Bastion Host (`22`번 포트) ➡️ 프라이빗 메인 EC2
> * **22번 포트 차단의 이유:** 배스천 호스트는 퍼블릭 서브넷에 떠 있는 가상 리눅스 컴퓨터입니다. 터미널 제어(SSH, 22번 포트)를 위해 열어두되, 해커들이 침투하지 못하도록 보안 그룹(SG) 설정을 통해 **"오직 우리 팀원들의 실제 공인 IP만 접근 허용"**하는 화이트리스트 빗장을 걸어 잠그는 것입니다.
</details>

<summary>🔍 Q6. 배스천 호스트는 포트번호대로 여러 개 존재하는 건가? (예: 80으로 요청되는 것만 오게 하거나 병렬 처리를 위해 여러 개를 두거나)</summary>

> **Q6. 배스천 호스트는 포트번호대로 여러 개 존재하는 건가? (예: 80으로 요청되는 것만 오게 하거나 병렬 처리를 위해 여러 개를 두거나)**
>
> **A6.** **아니오, 배스천 호스트는 오직 SSH(22번 포트) 통신 하나만 담당하며, 보통 인프라당 1대만 둡니다.** 일반 유저의 웹 요청(`80/443`)은 배스천을 거치지 않고 ALB로 가기 때문에 대량의 트래픽이 몰릴 일이 없습니다. 오직 개발팀 몇 명만 관리용으로 접속하므로, 사양이 가장 낮고 저렴한 EC2 컴퓨터 딱 1대만 켜놓고 공유 대문으로 쓰는 게 정석입니다.
</details>

<summary>🔍 Q7. 같은 EC2 컴퓨터여도 주인이 설치하는 프로그램에 따라 다른 용도로 쓸 수 있는 거지? 완전히 통제할 수 있는 컴퓨터가 여러 개 있고 용도는 알아서 쓰는 느낌으로 생각해도 되나?</summary>

<summary>🔍 Q9. 리눅스(Linux)라는 게 그 펭귄 그려져 있는 OS(운영체제)가 맞나? 일반 컴퓨터에도 리눅스를 깔 수 있는데 왜 서버 시장은 리눅스가 지배하고 있는 거지?</summary>

> ```

<summary>🔍 Q15. Public Subnet은 VPC 안에 있는 애인데, ALB가 왜 갑자기 AZ로 튀어나와?</summary>

> **Q15. Public Subnet은 VPC 안에 있는 애인데, ALB가 왜 갑자기 AZ로 튀어나와?**
>
> **A15.** ALB가 AZ 밖으로 튀어나가는 것이 아닙니다. 정확히는 **ALB가 여러 AZ에 있는 Public Subnet들에 걸쳐서 배치되는 것**입니다.
>
> 먼저 중요한 규칙이 있습니다.
>
> ```text
> 하나의 Subnet은 반드시 하나의 AZ에만 속한다.
> ```
>
> 그래서 고가용성 구조를 만들려면 AZ마다 Public Subnet과 Private Subnet을 따로 만듭니다.
>
> ```text
> AZ-1
> ├─ Public Subnet A
> └─ Private Subnet A
>
> AZ-2
> ├─ Public Subnet B
> └─ Private Subnet B
> ```
>
> ALB를 만들 때 AWS 콘솔에서 Public Subnet A와 Public Subnet B를 선택하면, ALB는 논리적으로 하나지만 실제로는 두 Public Subnet에 걸쳐서 동작합니다.
>
> ```text
> AZ-1 Public Subnet A ─┐
>                       ├─ 하나의 ALB처럼 동작
> AZ-2 Public Subnet B ─┘
> ```
>
> 즉, ALB가 AZ로 튀어나간 것이 아니라, Public Subnet 자체가 AZ별로 따로 있고, ALB가 그 Public Subnet들을 입구로 사용하는 구조입니다.
>
> 핵심 정리:
>
> ```text
> ALB = 여러 AZ의 Public Subnet에 걸쳐 있는 트래픽 입구
> ```

<summary>🔍 Q16. Region에 VPC가 전반적으로 있다는 게 무슨 뜻이야? VPC도 물리적으로 존재하는 거야?</summary>

> **Q16. Region에 VPC가 전반적으로 있다는 게 무슨 뜻이야? VPC도 물리적으로 존재하는 거야?**
>
> **A16.** VPC가 Region 전체에 물리적인 박스처럼 깔려 있다는 뜻은 아닙니다.
>
> Region과 AZ는 물리적 개념에 가깝습니다.
>
> ```text
> 서울 Region
> ├─ AZ-1: 실제 데이터센터 묶음
> ├─ AZ-2: 실제 데이터센터 묶음
> └─ AZ-3: 실제 데이터센터 묶음
> ```
>
> 반면 VPC는 **가상 네트워크 개념**입니다.
>
> 예를 들어 서울 Region에 VPC를 만든다는 것은 AWS가 이렇게 약속해주는 것입니다.
>
> ```text
> 서울 Region 안에서
> 10.0.0.0/16이라는 사설 IP 범위를
> 이 사용자의 가상 네트워크로 사용하게 해주겠다.
> ```
>
> 즉, VPC는 물리적인 장비가 아니라, 특정 Region 안에서 정의되는 가상의 사설 네트워크 공간입니다.
>
> 핵심은 이겁니다.
>
> ```text
> VPC는 Region에 속한다.
> Subnet은 AZ에 속한다.
> EC2는 Subnet 안에 생성된다.
> ```
>
> 더 짧게 정리하면:
>
> ```text
> Region / AZ = 물리적 위치 단위
> VPC = Region 단위의 가상 네트워크
> Subnet = VPC 일부를 특정 AZ에 배치한 것
> ```

</details>

<summary>🔍 Q17. 네이버 같은 서비스를 AZ-1, AZ-2로 나눠뒀을 때 AZ-1이 죽으면, AZ-2에 AZ-1이 가진 모든 정보가 있어야 하는 거 아니야?</summary>

> **Q17. 네이버 같은 서비스를 AZ-1, AZ-2로 나눠뒀을 때 AZ-1이 죽으면, AZ-2에 AZ-1이 가진 모든 정보가 있어야 하는 거 아니야?**
>
> **A17.** 큰 방향은 맞지만, 여기서 **서버가 가진 것**과 **데이터 저장소가 가진 것**을 구분해야 합니다.
>
> EC2 서버는 보통 중요한 유저 데이터를 직접 가지고 있으면 안 됩니다. EC2는 백엔드 로직을 실행하는 서버입니다.
>
> EC2가 가진 것:
>
> ```text
> 애플리케이션 코드
> API 처리 로직
> 인증/인가 로직
> DB에 요청하는 코드
> 설정값
> ```
>
> EC2가 직접 들고 있으면 안 되는 것:
>
> ```text
> 회원 정보
> 주문 내역
> 게시글
> 댓글
> 결제 정보
> 이미지 파일
> ```
>
> 그러므로 AZ-2의 EC2가 AZ-1 EC2의 모든 유저 데이터를 가지고 있어야 하는 것은 아닙니다.
>
> 정확히는:
>
> ```text
> AZ-2의 EC2도 같은 서버 로직을 실행하고,
> 필요한 데이터 저장소에 접근할 수 있어야 한다.
> ```
>
> 즉, ALB Failover는 “AZ-2 서버가 AZ-1 서버의 모든 데이터를 복사해서 가지고 있다”는 뜻이 아닙니다.
>
> 더 정확한 구조는 이겁니다.
>
> ```text
> 사용자
> ↓
> ALB
> ↓
> EC2 서버
> ↓
> 데이터 저장소
> ```
>
> 서버는 데이터를 직접 들고 있는 것이 아니라, 데이터가 있는 저장소에 요청해서 가져옵니다.
>
> 핵심 정리:
>
> ```text
> 서버는 같은 로직을 실행해야 하고,
> 데이터는 별도 저장소에서 관리되어야 한다.
> ```

</details>

<summary>🔍 Q18. 서버라는 곳은 데이터를 가진 곳이 아니라, 데이터를 요청하는 로직이 담긴 곳이라고 이해해도 돼?</summary>

> **Q18. 서버라는 곳은 데이터를 가진 곳이 아니라, 데이터를 요청하는 로직이 담긴 곳이라고 이해해도 돼?**
>
> **A18.** 네, 매우 좋은 이해입니다.
>
> EC2 같은 백엔드 서버는 보통 데이터를 영구 저장하는 곳이라기보다, 요청을 처리하는 로직이 실행되는 곳입니다.
>
> 예를 들어 배달앱 서버라면 EC2는 이런 일을 합니다.
>
> ```text
> 회원가입 요청 처리
> 로그인 요청 처리
> 가게 목록 조회 로직 실행
> 주문 생성 로직 실행
> 결제 요청 처리
> 리뷰 작성 처리
> ```
>
> 하지만 실제 회원 정보, 주문 정보, 리뷰 데이터 등을 EC2 안에 직접 저장하는 것은 좋지 않습니다. EC2가 죽으면 그 데이터에 접근할 수 없기 때문입니다.
>
> 그래서 EC2는 보통 이런 식으로 동작합니다.
>
> ```text
> 사용자 요청
> ↓
> EC2 서버
> ↓
> 데이터 저장소에 요청
> ↓
> 데이터 가져옴
> ↓
> 응답 생성
> ↓
> 사용자에게 반환
> ```
>
> 프로그래밍 비유로는 call by reference 느낌과 비슷하게 이해해도 됩니다.
>
> 다만 진짜 메모리 주소를 들고 있는 것은 아니고, EC2는 DB endpoint, 인증 정보, 네트워크 경로, 보안 그룹 권한 등을 통해 데이터 저장소에 접근합니다.
>
> 핵심 정리:
>
> ```text
> EC2 = 로직 실행
> 데이터 저장소 = 실제 데이터 보관
> ```

</details>

<summary>🔍 Q19. AZ-1의 Private Subnet A와 AZ-2의 Private Subnet B는 완전히 같은 내용을 담고 있어야 하는 거 아니야?</summary>

> **Q19. AZ-1의 Private Subnet A와 AZ-2의 Private Subnet B는 완전히 같은 내용을 담고 있어야 하는 거 아니야?**
>
> **A19.** 거의 맞는 감각이지만, 표현을 조금 고쳐야 합니다.
>
> Private Subnet A와 Private Subnet B 자체가 내용을 담는 것은 아닙니다. Subnet은 네트워크 구역입니다.
>
> 실제로 애플리케이션 로직을 가지고 실행하는 것은 Subnet 안의 EC2입니다.
>
> 따라서 정확한 표현은 다음과 같습니다.
>
> ```text
> AZ-1의 Private Subnet A 안에 있는 EC2 A와
> AZ-2의 Private Subnet B 안에 있는 EC2 B가
> 같은 애플리케이션 로직을 실행해야 한다.
> ```
>
> 예를 들면:
>
> ```text
> AZ-1 EC2 A → backend-app:v1.0.0 실행
> AZ-2 EC2 B → backend-app:v1.0.0 실행
> ```
>
> 이렇게 두 EC2가 같은 백엔드 버전을 실행해야 ALB가 어느 쪽으로 요청을 보내도 같은 결과를 낼 수 있습니다.
>
> 다만 이것은 DB처럼 실시간으로 서로 코드를 동기화하는 개념이 아닙니다.
>
> 서버 로직은 보통 배포 시스템을 통해 여러 서버에 동일하게 배포합니다.
>
> ```text
> GitHub 코드
> ↓
> Docker image 생성
> ↓
> 이미지 저장소에 업로드
> ↓
> AZ-1 EC2와 AZ-2 EC2가 같은 이미지 실행
> ```
>
> 핵심 정리:
>
> ```text
> Subnet이 같은 내용을 담는 것이 아니라,
> 각 Subnet 안의 EC2가 같은 앱 버전을 실행해야 한다.
> ```

<summary>🔍 Q20. DB 계층, RDS Multi-AZ, Aurora, Redis, S3는 뭐야?</summary>

> **Q20. DB 계층, RDS Multi-AZ, Aurora, Redis, S3는 뭐야?**
>
> **A20.** 여기서 말한 “DB 계층”은 더 정확히는 **데이터 저장소 계층**입니다.
>
> EC2가 데이터를 직접 들고 있지 않고, 데이터를 따로 저장하는 곳들을 말합니다.
>
> 대표적으로 다음과 같은 것들이 있습니다.
>
> ```text
> RDS / Aurora = 회원, 주문, 결제 같은 정형 데이터 저장
> Redis = 캐시, 세션, 임시 데이터 저장
> S3 = 이미지, 파일, 백업 저장
> ```
>
> 예를 들어 배달앱이라면:
>
> ```text
> 회원 정보 → RDS 또는 Aurora
> 주문 내역 → RDS 또는 Aurora
> 로그인 세션 → Redis
> 인기 가게 목록 캐시 → Redis
> 메뉴 사진 → S3
> 리뷰 사진 → S3
> ```
>
> 즉, EC2 서버는 요청을 처리하는 로직을 실행하고, 실제 데이터는 이런 데이터 저장소 계층에 맡깁니다.
>
> 다만 이 부분은 DB와 데이터 저장소 설계에 해당하므로, 현재 단계에서는 깊게 들어가기보다 다음 정도만 기억하면 됩니다.
>
> ```text
> EC2는 로직 실행
> 데이터는 별도 저장소에 보관
> ```

</details>

<summary>🔍 Q21. 지금까지 했던 내용을 DB 빼고 정리하면 어떻게 돼?</summary>

> **Q21. 지금까지 했던 내용을 DB 빼고 정리하면 어떻게 돼?**
>
> **A21.** DB를 제외하면 지금까지의 핵심 구조는 다음과 같습니다.
>
> ```text
> Region = 지리적 지역
> AZ = Region 안의 물리적으로 분리된 데이터센터 묶음
> VPC = 특정 Region 안에서 정의되는 가상 사설 네트워크
> Subnet = VPC의 IP 범위를 잘라 특정 AZ에 배치한 것
> Public Subnet = 인터넷과 만나는 구역
> Private Subnet = 백엔드 서버를 숨겨두는 내부 구역
> ALB = 사용자 요청을 받아 살아 있는 EC2로 분배하는 장치
> EC2 = 백엔드 애플리케이션 로직을 실행하는 서버
> ASG = 같은 EC2를 여러 AZ에 자동 배치하고 유지하는 장치
> ```
>
> 전체 구조는 다음과 같습니다.
>
> ```text
> 사용자
> ↓
> Internet
> ↓
> Internet Gateway
> ↓
> ALB
> ↓
> Private Subnet의 EC2 서버
> ```
>
> AZ별로 보면:
>
> ```text
> 서울 Region
> └─ VPC
>
>    AZ-1
>    ├─ Public Subnet A
>    │  └─ ALB 입구 A
>    └─ Private Subnet A
>       └─ EC2 A
>
>    AZ-2
>    ├─ Public Subnet B
>    │  └─ ALB 입구 B
>    └─ Private Subnet B
>       └─ EC2 B
> ```
>
> 장애 상황에서는:
>
> ```text
> AZ-1 장애 발생
> ↓
> AZ-1의 ALB 입구 A, EC2 A 장애
> ↓
> ALB가 Health Check로 EC2 A 장애 감지
> ↓
> EC2 A로 트래픽 보내지 않음
> ↓
> 살아 있는 AZ-2의 EC2 B로 요청 전달
> ```
>
> 단, 이게 가능하려면 EC2 B도 EC2 A와 같은 애플리케이션 로직을 실행하고 있어야 합니다.
>
> 핵심 정리:
>
> ```text
> 여러 AZ에 같은 서버 로직을 실행하는 EC2를 나눠두고,
> ALB가 살아 있는 EC2로 요청을 보내게 하는 구조가
> 서버 고가용성의 기본이다.
> ```

</details>

<summary>🔍 Q22. 왜 하나의 Subnet은 하나의 AZ에만 들어가게 해놓은 거야?</summary>

> **Q22. 왜 하나의 Subnet은 하나의 AZ에만 들어가게 해놓은 거야?**
>
> **A22.** Subnet은 단순한 IP 범위가 아니라, **그 IP 범위를 어느 AZ의 네트워크 공간에 배치할지까지 포함하는 개념**입니다.
>
> 예를 들어:
>
> ```text
> Private Subnet A: 10.0.2.0/24
> ```
>
> 이 말은 단순히 “10.0.2.x IP 대역”이라는 뜻만이 아닙니다.
>
> 정확히는:
>
> ```text
> 10.0.2.0/24 범위의 리소스들은 AZ-1에 배치된다.
> ```
>
> 라는 의미입니다.
>
> 만약 하나의 Subnet이 여러 AZ에 걸칠 수 있다면 장애 범위가 애매해집니다.
>
> 예를 들어 이런 구조가 가능하다고 해봅시다.
>
> ```text
> Private Subnet A: 10.0.2.0/24
>
> 일부 EC2는 AZ-1에 있음
> 일부 EC2는 AZ-2에 있음
> ```
>
> 이때 AZ-1 장애가 발생하면 Private Subnet A 안의 일부 서버는 죽고, 일부 서버는 살아 있게 됩니다.
>
> 그러면 “Private Subnet A가 장애 났다”라고 말하기도 애매하고, 장애 경계도 불명확해집니다.
>
> 반대로 Subnet이 하나의 AZ에만 속하면 장애 범위가 명확해집니다.
>
> ```text
> Private Subnet A → AZ-1
> Private Subnet B → AZ-2
> ```
>
> AZ-1 장애 발생 시:
>
> ```text
> Private Subnet A는 영향 받음
> Private Subnet B는 영향 없음
> ```
>
> 이렇게 장애 경계가 깔끔하게 나뉩니다.
>
> 핵심 정리:
>
> ```text
> Subnet을 하나의 AZ에만 묶는 이유는
> 물리적 위치와 장애 범위를 명확하게 분리하기 위해서다.
> ```
>
> 더 짧게 말하면:
>
> ```text
> VPC = Region 전체에서 쓸 수 있는 가상 네트워크
> Subnet = 그 VPC의 일부를 특정 AZ에 꽂아 넣은 것
> ```

</details>

<summary>🔍 Q23. 다른 AZ에 있는 서버끼리는 어떻게 동기화돼? 그냥 처음 앱을 배포할 때 여러 서버에 해놓는 느낌으로 이해해도 돼?</summary>

> **Q23. 다른 AZ에 있는 서버끼리는 어떻게 동기화돼? 그냥 처음 앱을 배포할 때 여러 서버에 해놓는 느낌으로 이해해도 돼?**
>
> **A23.** 네, 서버 로직에 대해서는 그렇게 이해하면 됩니다.
>
> 중요한 표현은 이것입니다.
>
> ```text
> 서버 로직은 실시간 동기화가 아니라 동일 배포에 가깝다.
> ```
>
> AZ-1의 EC2와 AZ-2의 EC2가 서로 계속 코드를 복사하면서 동기화하는 것이 아닙니다.
>
> 대신 배포 시스템이 같은 애플리케이션 버전을 두 서버에 모두 배포합니다.
>
> 예를 들어:
>
> ```text
> backend-app:v1.0.0
> ```
>
> 이 버전을 AZ-1과 AZ-2의 EC2가 둘 다 실행합니다.
>
> ```text
> AZ-1
> └─ EC2 A
>    └─ backend-app:v1.0.0 실행
>
> AZ-2
> └─ EC2 B
>    └─ backend-app:v1.0.0 실행
> ```
>
> 그러면 ALB가 어느 EC2로 요청을 보내도 같은 API 로직으로 처리할 수 있습니다.
>
> 앱 버전이 바뀌면 둘 다 새 버전으로 배포해야 합니다.
>
> ```text
> 기존 버전: backend-app:v1.0.0
> 새 버전: backend-app:v1.1.0
>
> AZ-1 EC2 A → backend-app:v1.1.0
> AZ-2 EC2 B → backend-app:v1.1.0
> ```
>
> 이 과정은 보통 다음과 같은 도구로 자동화합니다.
>
> ```text
> Docker image
> AMI
> Launch Template
> Auto Scaling Group
> CI/CD
> ECS / EKS
> CodeDeploy
> ```
>
> 핵심 정리:
>
> ```text
> 서버 코드 = 동일 배포
> 유저 데이터 = 별도 저장소에서 관리
> ```
>
> 즉:
>
> ```text
> AZ-1 EC2와 AZ-2 EC2가 서로 대화하면서 코드를 복사하는 게 아니라,
> 배포 시스템이 같은 코드를 둘 다에게 깔아주는 것이다.
> ```

</details>

<summary>🔍 Q24. DB는 진짜 실시간으로 여러 곳에 저장되고 복제되는 거야?</summary>

> **Q24. DB는 진짜 실시간으로 여러 곳에 저장되고 복제되는 거야? 서버는 특정 시점에 배포하고 다음 배포 전까지 유지되는 거잖아.**
>
> **A24.** 맞습니다. 서버와 DB는 “변하는 방식”이 다릅니다.
>
> 서버 코드는 보통 배포 시점에 바뀝니다.
>
> ```text
> 월요일 10:00 backend-app:v1.0.0 배포
> 수요일 15:00 backend-app:v1.1.0 배포
> ```
>
> 즉, 서버는 특정 시점에 같은 버전의 앱을 여러 EC2에 배포해두고, 다음 배포 전까지 그 버전으로 계속 요청을 처리합니다.
>
> 반면 DB 데이터는 사용자가 서비스를 쓰는 매 순간 바뀝니다.
>
> ```text
> 10:00:01 회원가입
> 10:00:03 로그인
> 10:00:05 주문 생성
> 10:00:08 결제 상태 변경
> 10:00:10 리뷰 작성
> ```
>
> 그래서 서버 코드는 **동일 배포**가 핵심이고, DB 데이터는 **지속적인 복제**가 핵심입니다.
>
> ```text
> 서버 코드 = 배포할 때 여러 서버에 같은 버전을 깔아둔다
> DB 데이터 = 운영 중 계속 바뀌므로 다른 AZ에도 계속 복제한다
> ```
>
> RDS Multi-AZ 같은 구성에서는 Primary DB에 쓰인 변경사항이 다른 AZ의 Standby DB로 거의 실시간에 가깝게 복제됩니다.
>
> 단, 물리적으로 완전히 같은 나노초에 두 군데에 동시에 저장된다는 뜻은 아닙니다. 더 정확히는:
>
> ```text
> Primary DB에 쓰기 발생
> ↓
> 다른 AZ의 Standby DB 또는 스토리지 계층에 복제
> ↓
> 필요한 조건을 만족하면 쓰기 성공 처리
> ```
>
> 핵심 정리:
>
> ```text
> 서버 = 같은 프로그램을 여러 곳에 미리 배포
> DB = 계속 바뀌는 데이터를 다른 곳에도 계속 복제
> ```

</details>

<summary>🔍 Q25. RDS는 뭐야? Redis랑 Aurora랑 S3도 나오던데 DB를 아예 몰라.</summary>

> **Q25. RDS는 뭐야? Redis랑 Aurora랑 S3도 나오던데 DB를 아예 몰라.**
>
> **A25.** 먼저 DB는 **Database**, 즉 데이터를 정리해서 저장하고 꺼내 쓰는 시스템입니다.
>
> 예를 들어 배달앱이라면 이런 데이터가 생깁니다.
>
> ```text
> 회원 정보
> 주문 내역
> 결제 상태
> 리뷰
> 가게 정보
> 메뉴 정보
> ```
>
> 이런 데이터를 EC2 서버 안의 파일에 대충 저장하면 위험하고 관리가 어렵기 때문에, 전문적으로 데이터를 저장하고 조회하는 DB를 사용합니다.
>
> 처음에는 관계형 DB를 엑셀 표 여러 개라고 생각하면 됩니다.
>
> ```text
> users 테이블
>
> id | name | email
> ---|------|-------------------------
> 1  | 민수 | minsu@example.com
> 2  | 지영 | jiyoung@example.com
> 3  | 찬혁 | cksrowldms9@g.hongik.ac.kr
> ```
>
> ```text
> orders 테이블
>
> id | user_id | menu   | price | status
> ---|---------|--------|-------|--------
> 1  | 3       | 치킨   | 23000 | 결제완료
> 2  | 3       | 피자   | 18000 | 배달중
> ```
>
> 대표적인 관계형 DB 엔진은 다음과 같습니다.
>
> ```text
> MySQL
> PostgreSQL
> MariaDB
> Oracle
> SQL Server
> ```
>
> **RDS**는 DB 엔진 이름이 아니라, AWS가 관계형 DB를 대신 운영해주는 관리형 서비스입니다.
>
> ```text
> RDS = Relational Database Service
> RDS = AWS가 MySQL/PostgreSQL 같은 DB를 대신 설치/운영해주는 서비스
> ```
>
> 직접 EC2에 MySQL을 설치하면 백업, 장애 대응, 보안 패치 등을 직접 해야 합니다.
>
> ```text
> EC2 생성
> ↓
> Linux 설치
> ↓
> MySQL 직접 설치
> ↓
> 백업 직접 설정
> ↓
> 장애 대응 직접 설정
> ↓
> 보안 패치 직접 관리
> ```
>
> RDS를 쓰면 AWS가 이런 운영 부담을 많이 대신 처리해줍니다.
>
> 비슷하게 나왔던 이름들은 역할이 다릅니다.
>
> ```text
> RDS
> = AWS가 MySQL/PostgreSQL 같은 관계형 DB를 대신 운영해주는 서비스
>
> Redis
> = 빠른 임시 저장소/캐시
>
> Aurora
> = AWS가 만든 고성능 관계형 DB 엔진
>
> S3
> = 이미지/파일 저장소
> ```
>
> 배달앱 기준으로 보면:
>
> ```text
> 회원 정보 → RDS
> 주문 내역 → RDS
> 결제 내역 → RDS
> 로그인 세션 → Redis
> 인기 가게 목록 캐시 → Redis
> 메뉴 사진 → S3
> 리뷰 사진 → S3
> ```
>
> 핵심 정리:
>
> ```text
> EC2 = 백엔드 로직 실행
> RDS = 회원/주문/결제 같은 진짜 데이터 저장
> Redis = 빠른 임시 저장소
> S3 = 파일 창고
> Aurora = AWS가 만든 고성능 DB 엔진
> ```

</details>

<summary>🔍 Q26. 고가용성이 뭐야?</summary>

> **Q26. 자꾸 고가용성이라는 단어가 나오는데 이게 뭐야?**
>
> **A26.** 고가용성은 영어로 **High Availability**, 줄여서 **HA**라고 합니다.
>
> 뜻은:
>
> ```text
> 장애가 나도 서비스가 최대한 계속 사용 가능하게 만드는 성질
> ```
>
> 즉, “절대 안 죽는 시스템”이라는 뜻은 아닙니다. 더 정확히는 장애가 나더라도 서비스 중단 시간을 줄이고, 다른 구성요소가 자동으로 이어받게 만드는 설계입니다.
>
> 예를 들어 고가용성이 낮은 구조:
>
> ```text
> EC2 1대
> DB 1대
> ↓
> 둘 중 하나만 죽어도 서비스 전체 장애
> ```
>
> 고가용성을 고려한 구조:
>
> ```text
> EC2 여러 대
> ALB로 분산
> 여러 AZ에 배치
> DB도 Multi-AZ 구성
> ↓
> 하나가 죽어도 다른 쪽이 이어받음
> ```
>
> 고가용성에서 중요한 개념은 **SPOF**입니다.
>
> ```text
> SPOF = Single Point of Failure
> ```
>
> 뜻은:
>
> ```text
> 이것 하나 죽으면 전체 서비스가 죽는 지점
> ```
>
> 예:
>
> ```text
> EC2가 1대뿐이다
> → 그 EC2가 죽으면 서비스 죽음
>
> DB가 1대뿐이다
> → 그 DB가 죽으면 서비스 죽음
>
> ALB가 한 AZ에만 있다
> → 그 AZ가 죽으면 입구도 죽음
> ```
>
> 고가용성 설계는 이런 단일 장애 지점을 줄이는 방향입니다.
>
> ```text
> EC2 한 대만 두지 말자
> → 여러 AZ에 여러 EC2 배치
>
> DB 한 대만 두지 말자
> → Multi-AZ 구성
>
> 트래픽 입구도 한 곳에만 두지 말자
> → ALB를 여러 Public Subnet에 걸치게 함
> ```
>
> 핵심 정리:
>
> ```text
> 고가용성 = 하나가 죽어도 다른 쪽이 이어받아 서비스가 계속되게 만드는 설계
> ```

</details>

<details>
<summary>🔍 Q27. 일반 RDS Multi-AZ랑 Aurora는 뭐가 달라?</summary>

> **Q27. 일반 RDS Multi-AZ랑 Aurora 방식이랑 뭐가 다른지 잘 모르겠어.**
>
> **A27.** 둘 다 여러 AZ에 데이터를 복제해서 장애에 대비한다는 점에서는 비슷합니다. 하지만 구조가 다릅니다.
>
> 일반 RDS Multi-AZ는 다음과 같이 이해하면 쉽습니다.
>
> ```text
> AZ-1
> └─ Primary DB
>
> AZ-2
> └─ Standby DB
> ```
>
> 평소에는 Primary DB가 실제로 일합니다.
>
> ```text
> EC2 → Primary DB
> ```
>
> Standby DB는 다른 AZ에서 복제본으로 대기합니다.
>
> ```text
> Primary DB 변경사항
> ↓
> Standby DB에 복제
> ```
>
> 장애가 나면:
>
> ```text
> Primary DB 장애
> ↓
> Standby DB가 새 Primary로 승격
> ```
>
> 즉, 일반 RDS Multi-AZ는:
>
> ```text
> 일하는 DB 한 세트
> + 다른 AZ에 대기 DB 한 세트
> ```
>
> 느낌입니다.
>
> 반면 Aurora는 구조가 다릅니다.
>
> Aurora는 DB 엔진과 저장소를 분리해서 생각해야 합니다.
>
> ```text
> Aurora Writer Instance
> = SQL 처리하는 DB 엔진
>
> Aurora Cluster Volume
> = 여러 AZ에 복제된 공유 저장소
> ```
>
> 그림으로 보면:
>
> ```text
>          Aurora Writer Instance
>                   │
>                   ▼
>          Aurora Cluster Volume
>                   │
>      ┌────────────┼────────────┐
>      ▼            ▼            ▼
>     AZ-1         AZ-2         AZ-3
>   storage      storage      storage
>   copies       copies       copies
> ```
>
> 일반 RDS Multi-AZ:
>
> ```text
> Primary DB + Standby DB
> ```
>
> Aurora:
>
> ```text
> Writer/Reader Instance + 여러 AZ에 복제된 Cluster Volume
> ```
>
> 핵심 차이:
>
> ```text
> 일반 RDS Multi-AZ
> = DB 인스턴스와 저장소가 한 세트로 묶여 있고,
>   그 세트를 다른 AZ에 하나 더 둔다.
>
> Aurora
> = DB 엔진과 저장소를 분리하고,
>   저장소 계층 자체를 여러 AZ에 복제한다.
> ```
>
> 한 줄 정리:
>
> ```text
> RDS Multi-AZ = Primary와 Standby의 주종 구조
> Aurora = Writer/Reader 인스턴스 + 여러 AZ에 복제된 공유 저장소 구조
> ```

</details>

<details>
<summary>🔍 Q28. Aurora가 분산되어 있다는 건 Primary를 조각조각 떼어놓는다는 뜻이야?</summary>

> **Q28. Aurora가 분산되어 있다고 했는데, 그러면 온전한 Primary를 조각조각 떼어놓는다는 거야? AZ끼리 서로 다른 정보를 갖고 있다는 건가?**
>
> **A28.** 아닙니다. Aurora가 데이터를 여러 AZ에 분산 저장한다는 말은, AZ마다 서로 다른 정보만 갖는다는 뜻이 아닙니다.
>
> 잘못된 이해:
>
> ```text
> 전체 DB = A + B + C
>
> AZ-1에는 A만 있음
> AZ-2에는 B만 있음
> AZ-3에는 C만 있음
> ```
>
> 이러면 AZ-1이 죽었을 때 A 조각이 사라지기 때문에 위험합니다.
>
> Aurora에 더 가까운 이해:
>
> ```text
> 전체 DB = A + B + C
>
> A 조각의 복사본 → 여러 AZ에 존재
> B 조각의 복사본 → 여러 AZ에 존재
> C 조각의 복사본 → 여러 AZ에 존재
> ```
>
> 즉:
>
> ```text
> 조각으로 나누기는 하지만,
> 각 조각을 하나만 두는 게 아니라 여러 AZ에 복제해둔다.
> ```
>
> 여기서 중요한 점은 Primary DB Instance가 조각나는 것이 아니라는 점입니다.
>
> ```text
> Primary DB Instance
> = SQL 요청을 받고 쓰기 작업을 처리하는 DB 엔진
>
> Aurora Cluster Volume
> = 여러 AZ에 복제되어 있는 공유 저장소
> ```
>
> 분산되는 것은 주로 데이터가 저장되는 **스토리지 계층**입니다.
>
> ```text
> 분산되는 것
> = DB 데이터가 저장되는 스토리지 블록들
>
> 분산되는 것이 아닌 것
> = Primary DB Instance 자체
> ```
>
> 핵심 정리:
>
> ```text
> Primary가 조각나는 것이 아니라,
> Primary가 사용하는 저장소가 여러 AZ에 복제되어 있는 것이다.
> ```

</details>

<details>
<summary>🔍 Q29. 그러면 RDS Multi-AZ랑 Aurora는 쪼갠다는 것 말고 뭐가 달라?</summary>

> **Q29. RDS Multi-AZ랑 Aurora의 차이가 그냥 Aurora는 쪼갠다는 것밖에 없어 보이는데?**
>
> **A29.** 겉으로 보면 둘 다 여러 AZ에 복제하기 때문에 비슷해 보입니다.
>
> 하지만 차이는 단순히 “쪼갠다”가 아니라:
>
> ```text
> DB 엔진과 저장소를 붙여놓느냐,
> DB 엔진과 저장소를 분리하느냐
> ```
>
> 입니다.
>
> 일반 RDS Multi-AZ:
>
> ```text
> AZ-1
> └─ Primary DB Instance
>    ├─ DB 엔진
>    └─ 저장소
>
> AZ-2
> └─ Standby DB Instance
>    ├─ DB 엔진
>    └─ 복제된 저장소
> ```
>
> 느낌:
>
> ```text
> 일하는 DB 한 세트
> + 다른 AZ에 대기 DB 한 세트
> ```
>
> Aurora:
>
> ```text
> Writer Instance
> Reader Instance
> ↓
> Shared Cluster Volume
> ↓
> 여러 AZ의 Storage Node들
> ```
>
> 느낌:
>
> ```text
> DB 인스턴스들은 엔진 역할을 하고,
> 실제 영구 데이터는 공유 Cluster Volume에 있음
> ```
>
> 이 구조 차이 때문에 생기는 차이:
>
> ```text
> 일반 RDS Multi-AZ
> = Primary DB가 일하고 Standby DB가 장애 대기
>
> Aurora
> = Writer/Reader Instance가 공유 Cluster Volume에 붙어서 사용
> ```
>
> 한 줄 정리:
>
> ```text
> 일반 RDS Multi-AZ
> = DB 서버 한 세트를 다른 AZ에 하나 더 복제해둔다.
>
> Aurora
> = DB 서버와 저장소를 분리하고, 저장소 자체를 여러 AZ에 복제해둔다.
> ```

</details>

<details>
<summary>🔍 Q30. Aurora는 공유 저장소가 있고 거기서 동시에 다운받는 느낌이야? Primary도 Cluster Volume에서 내려받는 거야?</summary>

> **Q30. RDS는 인스턴스를 전송하는 느낌이라면, Aurora는 공유 저장소가 있고 거기서 동시에 다운받는 느낌이야? Primary도 Cluster Volume에서 내려받는 거야?**
>
> **A30.** 방향은 맞지만, “다운받는다”보다는 **붙어서 읽고 쓴다**가 더 정확합니다.
>
> Aurora에서는 Primary도 Cluster Volume을 사용합니다.
>
> ```text
> Aurora DB Cluster
>
> Writer Instance / Primary
> = 읽기 + 쓰기 담당
>
> Reader Instance / Aurora Replica
> = 읽기 담당
>
> Cluster Volume
> = 실제 데이터가 저장되는 공유 저장소
> ```
>
> 그림으로 보면:
>
> ```text
>                  Application
>                      ↓
>               Writer Endpoint
>                      ↓
>            Primary / Writer Instance
>                      ↓
>               Cluster Volume
>       ┌──────────────┼──────────────┐
>       ↓              ↓              ↓
>      AZ-1           AZ-2           AZ-3
>   storage copies  storage copies  storage copies
> ```
>
> Primary가 데이터를 통째로 다운로드해서 자기 안에 들고 있는 것은 아닙니다.
>
> 더 정확히는:
>
> ```text
> Primary DB Instance가 Cluster Volume에 연결되어 있고,
> 필요한 데이터 page를 읽고,
> 변경사항을 Cluster Volume에 쓴다.
> ```
>
> Primary 안에도 메모리 캐시나 임시 데이터는 있을 수 있습니다.
>
> ```text
> Primary Instance 안에 있을 수 있는 것
> - DB 엔진
> - 쿼리 처리 로직
> - 연결 관리
> - 트랜잭션 처리
> - 메모리 캐시
> - 임시 작업 공간
> ```
>
> 하지만 영구 데이터 원본은 Cluster Volume 쪽에 있다고 이해하면 됩니다.
>
> 비유:
>
> ```text
> Cluster Volume = 도서관 서고
> Primary Instance = 사서 + 책상
> 캐시 = 사서 책상 위에 잠깐 올려둔 자주 보는 책
> ```
>
> 핵심 정리:
>
> ```text
> 다운로드한다 ❌
> Cluster Volume에 붙어서 읽고 쓴다 ⭕
> ```

</details>

<details>
<summary>🔍 Q31. Aurora DB에서 글 하나가 추가되면 실제로 어떻게 돼?</summary>

> **Q31. Aurora DB를 쓴다고 치자. 글이 하나 추가됐어. 그러면 어떻게 되는 거야?**
>
> **A31.** 예를 들어 게시판 앱에서 사용자가 글을 썼다고 해봅시다.
>
> ```text
> 제목: 안녕하세요
> 내용: 첫 글입니다
> 작성자: 찬혁
> ```
>
> 백엔드 서버는 Aurora DB에 이런 SQL 요청을 보냅니다.
>
> ```sql
> INSERT INTO posts (title, content, user_id)
> VALUES ('안녕하세요', '첫 글입니다', 7);
> ```
>
> 전체 흐름:
>
> ```text
> 사용자
> ↓
> ALB
> ↓
> EC2 백엔드 서버
> ↓
> Aurora Writer Instance
> ↓
> Aurora Cluster Volume
> ↓
> 여러 AZ의 Storage Node에 복제
> ```
>
> 단계별로 보면:
>
> ```text
> 1. 사용자가 글 작성 버튼을 누름
>
> 2. 요청이 ALB를 거쳐 EC2 백엔드 서버로 감
>
> 3. EC2가 Aurora Writer Instance에게 INSERT 요청을 보냄
>
> 4. Writer Instance가 SQL을 해석함
>    "posts 테이블에 새 row를 넣어야겠네."
>    "인덱스도 갱신해야겠네."
>    "트랜잭션 로그도 남겨야겠네."
>
> 5. Writer Instance가 변경사항을 Cluster Volume에 기록함
>
> 6. Cluster Volume을 구성하는 storage layer가
>    여러 AZ의 Storage Node에 변경사항을 복제함
>
> 7. 충분히 저장되면 쓰기 성공 응답
>
> 8. EC2가 사용자에게 "글 작성 완료" 응답
> ```
>
> 중요한 점:
>
> ```text
> Writer Instance = SQL 처리 엔진
> Cluster Volume = 영구 데이터 저장소
> Storage Node = 실제 데이터 page 복사본이 저장되는 저장 계층
> ```
>
> 핵심 정리:
>
> ```text
> EC2가 Writer에게 "글 저장해줘"라고 말함
> → Writer가 SQL을 처리함
> → 실제 글 데이터는 Cluster Volume에 저장됨
> → Cluster Volume을 구성하는 storage layer가 여러 AZ에 복제함
> → 저장 완료
> ```

</details>

<details>
<summary>🔍 Q32. Aurora에서 Storage도 Primary를 정해야 하는 거 아니야?</summary>

> **Q32. Cluster Volume이 대장이고 밑에 있는 Storage에 전달되는 건 알겠어. 그런데 어차피 다 쓰진 않을 거 아니야. 어떤 AZ의 Storage를 Primary로 두는 개념이 있어야 하는 거 아니야?**
>
> **A32.** Aurora에서는 **특정 AZ의 Storage를 Primary로 둔다**고 이해하지 않는 것이 좋습니다.
>
> Aurora에서 Primary 역할은 Storage가 아니라 **Writer DB Instance** 쪽에 있습니다.
>
> ```text
> 쓰기 대장 = Writer / Primary DB Instance
> 저장소 대장 = 특정 AZ의 Storage가 아니라 Aurora Cluster Volume 전체
> ```
>
> 잘못된 이해:
>
> ```text
> AZ-1 Storage = Primary Storage
> AZ-2 Storage = Replica Storage
> AZ-3 Storage = Replica Storage
> ```
>
> Aurora에 더 가까운 이해:
>
> ```text
> Aurora Cluster Volume
> ├─ AZ-1 storage nodes
> ├─ AZ-2 storage nodes
> └─ AZ-3 storage nodes
> ```
>
> 사용자나 DB 인스턴스 입장에서는 이 전체가 하나의 논리적 저장소처럼 보입니다.
>
> 쓰기 요청은 Writer DB Instance로 갑니다.
>
> ```text
> EC2
> ↓
> Aurora Writer Instance
> ↓
> Aurora Cluster Volume
> ↓
> 여러 AZ의 Storage Nodes
> ```
>
> 즉:
>
> ```text
> Primary 역할은 Storage Node가 아니라 Writer Instance가 맡는다.
> ```
>
> Storage 쪽은 특정 AZ 하나가 원본이고 나머지가 복사본이라는 느낌이 아니라, Cluster Volume 전체가 여러 AZ의 Storage Node로 구성된 하나의 논리적 저장소처럼 동작합니다.
>
> 핵심 정리:
>
> ```text
> Aurora에서 Primary는 Storage가 아니라 Writer Instance 쪽에 있고,
> Storage는 특정 AZ 하나가 대장이 아니라 Cluster Volume 전체가 관리한다.
> ```

</details>

<details>
<summary>🔍 Q33. 데이터를 읽을 때는 모든 AZ Storage를 들르는 거야? 그냥 Cluster Volume에 들린다는 거야?</summary>

> **Q33. 데이터를 읽어오는 상황에서는 어떻게 돼? 데이터를 사용하려면 모든 AZ에 들리진 않을 거 아니야. 그냥 Cluster Volume에 들리는 거야?**
>
> **A33.** 읽을 때 EC2가 AZ-1 Storage, AZ-2 Storage, AZ-3 Storage를 직접 고르는 것이 아닙니다.
>
> EC2는 Aurora의 Writer 또는 Reader DB Instance에 SQL 요청을 보냅니다.
>
> ```text
> EC2 → Aurora Reader 또는 Writer Instance → Cluster Volume
> ```
>
> 예를 들어 게시글 목록을 조회한다고 하면:
>
> ```sql
> SELECT * FROM posts WHERE user_id = 7;
> ```
>
> 흐름은:
>
> ```text
> 사용자
> ↓
> ALB
> ↓
> EC2 백엔드 서버
> ↓
> Aurora Reader Instance 또는 Writer Instance
> ↓
> Aurora Cluster Volume
> ↓
> 결과 반환
> ```
>
> 읽기 요청을 모든 AZ의 Storage Node에 직접 다 보내는 느낌이 아닙니다.
>
> 사용자 관점에서는:
>
> ```text
> Reader Instance가 Cluster Volume에서 읽는다.
> ```
>
> 라고 이해하면 됩니다.
>
> 내부 구현 관점에서는:
>
> ```text
> Cluster Volume은 여러 AZ의 Storage Node들로 구성되어 있으므로,
> 실제 물리적인 읽기는 그 Storage Node들 중 일부에서 일어난다.
> ```
>
> Reader Instance가 있으면 SELECT 같은 읽기 요청은 Reader Instance로 보낼 수 있습니다.
>
> ```text
> EC2
> ↓
> Aurora Reader Endpoint
> ↓
> Reader Instance
> ↓
> Cluster Volume에서 읽기
> ```
>
> 핵심 정리:
>
> ```text
> EC2가 Storage Node를 직접 찾아가지 않는다.
> EC2는 DB Instance에 SQL 요청을 보낸다.
> DB Instance가 Cluster Volume에서 필요한 데이터를 읽는다.
> ```

</details>

<details>
<summary>🔍 Q34. Cluster Volume에 이미 데이터가 있다면 왜 Storage Node에서 읽는다고 해? 그리고 “조립한다”는 건 Storage마다 다른 데이터를 가진다는 뜻 아니야?</summary>

> **Q34. “Cluster Volume / Storage Layer가 내부 Storage Node 중 적절한 곳에서 데이터 page를 읽어온다”는 게 이해가 안 돼. Cluster Volume에 이미 내용이 있다면 왜 Storage로 또 가? 그리고 “조립한다”는 표현은 Storage Node마다 다른 내용을 갖고 있다는 뜻 아니야?**
>
> **A34.** 여기서 가장 중요한 교정은 이것입니다.
>
> ```text
> Cluster Volume은 Storage Node 위에 따로 있는 물리 저장소가 아니다.
> Cluster Volume은 여러 Storage Node들을 묶어서 하나의 저장소처럼 보이게 만든 논리적/가상 볼륨이다.
> ```
>
> 잘못된 이해:
>
> ```text
> Cluster Volume
> └─ 데이터 있음
>    ↓
> Storage Node
> └─ 또 데이터 있음
> ```
>
> 올바른 이해:
>
> ```text
> Cluster Volume
> = Storage Node들의 묶음을 하나의 볼륨처럼 보여주는 이름
>
> 실제 데이터 page/복사본은
> 여러 AZ의 Storage Node들에 저장됨
> ```
>
> 즉:
>
> ```text
> Cluster Volume에 데이터가 있다
> ```
>
> 라는 말은 정확히 풀면:
>
> ```text
> Cluster Volume을 구성하는 여러 Storage Node들에 데이터가 저장되어 있다
> ```
>
> 라는 뜻입니다.
>
> 그리고 “적절한 Storage Node에서 읽는다”는 말은 EC2나 Reader Instance가 직접 AZ Storage를 고른다는 뜻이 아닙니다.
>
> 정확히는:
>
> ```text
> Reader Instance가 Cluster Volume이라는 논리적 저장소에 read 요청을 보냄
> ↓
> Aurora storage layer 내부에서 실제 데이터를 가진 Storage Node들 중 사용 가능한 복제본을 통해 데이터를 제공
> ```
>
> 또한 “조립한다”는 표현은 Storage Node마다 서로 다른 컬럼이나 테이블을 갖고 있어서 퍼즐처럼 맞춘다는 뜻이 아닙니다.
>
> 잘못된 이해:
>
> ```text
> AZ-1 Storage에서 제목 가져오고
> AZ-2 Storage에서 내용 가져오고
> AZ-3 Storage에서 작성자 가져와서
> Reader가 조립한다 ❌
> ```
>
> 정확한 의미:
>
> ```text
> DB 엔진이 읽어온 data page를 해석해서
> SQL 결과 row 형태로 만들어준다 ⭕
> ```
>
> 예를 들어:
>
> ```sql
> SELECT title, content
> FROM posts
> WHERE user_id = 7;
> ```
>
> 이 요청이 오면 Reader Instance는:
>
> ```text
> posts 테이블에서 user_id = 7인 row를 찾고
> 필요한 column인 title, content를 뽑고
> 조건에 맞는 결과 목록을 만들어서 반환
> ```
>
> 이 과정을 “SQL 결과로 조립한다”고 표현한 것입니다.
>
> 여기서 data page는 DB가 내부적으로 읽는 저장 단위입니다.
>
> ```text
> Data Page 100번
> ├─ posts row id=1
> ├─ posts row id=2
> └─ posts row id=3
> ```
>
> DB는 보통 row 하나를 직접 디스크에서 쏙 꺼내기보다, 그 row가 들어 있는 data page를 읽고, 그 안에서 필요한 row와 column을 찾아냅니다.
>
> 따라서 “조립”의 의미는:
>
> ```text
> 여러 AZ Storage에서 서로 다른 내용을 모아 조립한다 ❌
>
> 읽어온 page에서 필요한 row/column을 뽑아
> SQL 결과표로 만드는 것 ⭕
> ```
>
> 최종 정리:
>
> ```text
> Cluster Volume
> = 물리 저장소가 따로 있는 곳이 아니라,
>   여러 Storage Node를 묶어서 만든 가상 저장소 이름
>
> Storage Node
> = 실제 데이터 page 복사본이 저장되는 곳
>
> Reader Instance
> = page를 읽어서 SQL 결과로 해석하는 DB 엔진
> ```
>
> 한 줄 요약:
>
> ```text
> Reader는 Cluster Volume에 읽기 요청을 하고,
> Cluster Volume은 내부 Storage Node들의 묶음이기 때문에
> 실제 데이터는 그 안의 복제본에서 읽힌다.
>
> “조립”은 AZ별 조각 맞추기가 아니라 SQL 결과를 만드는 과정이다.
> ```

</details>

<summary>🔍 Q35. ASG는 뭐야?</summary>

> **Q35. ASG는 뭐야?**
>
> **A35.** ASG는 **Auto Scaling Group**의 줄임말입니다.
>
> 한 줄로 말하면:
>
> ```text
> ASG = 같은 EC2 서버들을 자동으로 만들고, 유지하고, 필요하면 늘리거나 줄이는 AWS 기능
> ```
>
> 예를 들어 우리가 이런 구조를 만든다고 해봅시다.
>
> ```text
> AZ-1
> └─ Private Subnet A
>    └─ EC2 A
>
> AZ-2
> └─ Private Subnet B
>    └─ EC2 B
> ```
>
> 여기서 EC2 A와 EC2 B는 같은 백엔드 앱을 실행해야 합니다.
>
> ```text
> EC2 A → backend-app:v1.0.0
> EC2 B → backend-app:v1.0.0
> ```
>
> 그런데 이 EC2들을 사람이 하나하나 직접 만들고, 장애가 나면 직접 다시 만들면 너무 번거롭고 위험합니다.
>
> ASG는 이걸 자동으로 해줍니다.
>
> ```text
> EC2가 부족함
> ↓
> ASG가 새 EC2 생성
>
> EC2가 죽음
> ↓
> ASG가 새 EC2 생성
>
> 트래픽이 많아짐
> ↓
> ASG가 EC2 개수 증가
>
> 트래픽이 줄어듦
> ↓
> ASG가 EC2 개수 감소
> ```
>
> 즉, ASG는 EC2들을 자동으로 관리하는 장치입니다.
>
> 핵심 정리:
>
> ```text
> ASG = EC2 자동 생성 / 유지 / 확장 기능
> ```

</details>

<summary>🔍 Q36. ASG는 왜 필요한 거야?</summary>

> **Q36. ASG는 왜 필요한 거야? EC2를 그냥 직접 여러 개 만들면 안 돼?**
>
> **A36.** EC2를 직접 여러 개 만들 수도 있습니다. 하지만 실무에서는 사람이 직접 관리하면 문제가 많습니다.
>
> 예를 들어 EC2 2대를 직접 만들었다고 해봅시다.
>
> ```text
> AZ-1 EC2 A
> AZ-2 EC2 B
> ```
>
> 그런데 EC2 A가 죽으면 어떻게 해야 할까요?
>
> ```text
> EC2 A 장애
> ↓
> 사람이 콘솔 들어감
> ↓
> 새 EC2 생성
> ↓
> 보안 그룹 설정
> ↓
> 앱 설치
> ↓
> ALB에 연결
> ```
>
> 이걸 사람이 직접 하면 느리고 실수할 가능성도 큽니다.
>
> ASG를 쓰면 이렇게 됩니다.
>
> ```text
> EC2 A 장애
> ↓
> ASG가 장애 감지
> ↓
> 새 EC2 자동 생성
> ↓
> ALB Target Group에 연결
> ↓
> 서비스 복구
> ```
>
> 즉, ASG의 핵심 목적은 다음과 같습니다.
>
> ```text
> 1. 원하는 EC2 개수를 유지한다.
> 2. 장애 난 EC2를 자동으로 교체한다.
> 3. 트래픽에 따라 EC2 개수를 자동 조절한다.
> 4. 여러 AZ에 EC2를 나눠 배치해서 고가용성을 높인다.
> ```
>
> 핵심 정리:
>
> ```text
> ASG는 사람이 EC2를 직접 관리하지 않아도 되게 해주는 자동화 장치다.
> ```

</details>

<details>
<summary>🔍 Q37. ASG는 EC2 개수를 어떻게 유지해?</summary>

> **Q37. ASG는 EC2 개수를 어떻게 유지해?**
>
> **A37.** ASG에는 보통 세 가지 개수 설정이 있습니다.
>
> ```text
> 최소 개수 = 최소한 이만큼은 유지
> 희망 개수 = 평소에 유지하고 싶은 개수
> 최대 개수 = 아무리 늘어나도 여기까지만 허용
> ```
>
> 예를 들어 이렇게 설정할 수 있습니다.
>
> ```text
> 최소 개수: 2
> 희망 개수: 2
> 최대 개수: 4
> ```
>
> 이 뜻은:
>
> ```text
> 평소에는 EC2 2대를 유지하고,
> 트래픽이 많아지면 최대 4대까지 늘릴 수 있으며,
> 최소한 2대 밑으로는 내려가지 않게 하겠다.
> ```
>
> 만약 EC2 한 대가 죽으면:
>
> ```text
> 원래 희망 개수 = 2대
> 현재 살아 있는 EC2 = 1대
> ↓
> ASG가 새 EC2 1대 생성
> ↓
> 다시 EC2 2대 유지
> ```
>
> 그래서 ASG는 단순히 “늘리는 기능”만 있는 것이 아니라, **원하는 개수를 계속 유지하는 기능**도 합니다.
>
> 핵심 정리:
>
> ```text
> ASG는 설정된 희망 개수에 맞춰 EC2 수를 계속 유지한다.
> ```

</details>

<summary>🔍 Q38. ASG는 EC2를 만들 때 뭘 보고 똑같이 만들어?</summary>

> **Q38. ASG는 EC2를 만들 때 뭘 보고 똑같이 만들어?**
>
> **A38.** ASG는 보통 **Launch Template**이라는 설계도를 보고 EC2를 만듭니다.
>
> Launch Template은 EC2 생성 설명서라고 보면 됩니다.
>
> Launch Template에는 이런 정보가 들어갑니다.
>
> ```text
> 어떤 AMI를 사용할지
> 어떤 인스턴스 타입을 사용할지
> 어떤 보안 그룹을 붙일지
> 어떤 키페어를 사용할지
> 부팅할 때 어떤 스크립트를 실행할지
> 어떤 앱을 실행할지
> ```
>
> 예를 들어 백엔드 서버용 Launch Template은 이런 느낌입니다.
>
> ```text
> Launch Template
> - Amazon Linux 사용
> - t3.micro 사용
> - backend-server-sg 보안 그룹 사용
> - 부팅 시 Docker 설치
> - backend-app:v1.0.0 컨테이너 실행
> ```
>
> 그러면 ASG는 이 설계도를 보고 같은 형태의 EC2를 계속 만들 수 있습니다.
>
> ```text
> ASG
> ↓
> Launch Template 확인
> ↓
> 같은 설정의 EC2 생성
> ```
>
> 그래서 ASG가 새 EC2를 만들어도 기존 EC2와 같은 백엔드 앱을 실행할 수 있습니다.
>
> 핵심 정리:
>
> ```text
> Launch Template = ASG가 EC2를 만들 때 참고하는 설계도
> ASG = 그 설계도를 보고 EC2를 자동 생성하는 관리자
> ```

</details>

<summary>🔍 Q39. ASG랑 ALB는 어떻게 같이 동작해?</summary>

> **Q39. ASG랑 ALB는 어떻게 같이 동작해?**
>
> **A39.** ALB와 ASG는 역할이 다릅니다.
>
> ```text
> ALB = 사용자 요청을 살아 있는 EC2로 분산하는 장치
> ASG = EC2를 자동으로 만들고 유지하는 장치
> ```
>
> 같이 쓰면 구조는 이렇게 됩니다.
>
> ```text
> 사용자
> ↓
> ALB
> ↓
> ASG가 관리하는 EC2들
> ```
>
> ASG가 새 EC2를 만들면, 그 EC2는 ALB의 Target Group에 등록될 수 있습니다.
>
> ```text
> ASG가 EC2 생성
> ↓
> ALB Target Group에 등록
> ↓
> ALB가 Health Check 수행
> ↓
> 정상 상태면 트래픽 전달
> ```
>
> EC2가 죽으면:
>
> ```text
> EC2 장애
> ↓
> ALB가 Health Check 실패 감지
> ↓
> ALB가 해당 EC2로 트래픽 중단
> ↓
> ASG가 새 EC2 생성
> ↓
> 새 EC2가 Target Group에 등록
> ↓
> ALB가 새 EC2로 트래픽 전달
> ```
>
> 즉:
>
> ```text
> ALB는 요청을 어디로 보낼지 결정하고,
> ASG는 보낼 대상인 EC2들을 자동으로 유지한다.
> ```
>
> 핵심 정리:
>
> ```text
> ALB = 트래픽 분산
> ASG = 서버 자동 생성/유지
> ```

</details>

<summary>🔍 Q40. ASG는 고가용성과 어떤 관련이 있어?</summary>

> **Q40. ASG는 고가용성과 어떤 관련이 있어?**
>
> **A40.** 고가용성은 장애가 나도 서비스가 계속되게 만드는 설계입니다.
>
> ASG는 서버 계층의 고가용성을 만드는 데 중요한 역할을 합니다.
>
> 예를 들어 ASG가 여러 AZ의 Private Subnet을 대상으로 설정되어 있다고 해봅시다.
>
> ```text
> ASG 대상 Subnet
> - AZ-1 Private Subnet A
> - AZ-2 Private Subnet B
> ```
>
> 그러면 ASG는 EC2를 여러 AZ에 나눠서 띄울 수 있습니다.
>
> ```text
> AZ-1
> └─ EC2 A
>
> AZ-2
> └─ EC2 B
> ```
>
> AZ-1의 EC2가 죽으면:
>
> ```text
> EC2 A 장애
> ↓
> ALB가 EC2 A로 트래픽 중단
> ↓
> AZ-2의 EC2 B가 요청 처리
> ↓
> ASG가 새 EC2 생성
> ```
>
> 즉, ASG는 다음 역할을 통해 고가용성을 높입니다.
>
> ```text
> 1. 여러 AZ에 EC2 배치
> 2. 죽은 EC2 자동 교체
> 3. 트래픽 증가 시 EC2 자동 확장
> ```
>
> 하지만 중요한 점은 ASG가 데이터까지 복제해주는 것은 아니라는 점입니다.
>
> ```text
> ASG가 관리하는 것 = EC2 서버
> ASG가 관리하지 않는 것 = DB 데이터
> ```
>
> DB 데이터의 고가용성은 RDS Multi-AZ, Aurora 같은 DB 계층에서 따로 처리해야 합니다.
>
> 핵심 정리:
>
> ```text
> ASG는 서버 계층의 고가용성을 높이는 기능이다.
> DB 데이터 복제는 ASG가 아니라 DB 계층이 담당한다.
> ```

</details>
