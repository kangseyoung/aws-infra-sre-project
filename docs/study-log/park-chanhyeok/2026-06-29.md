<details>
<summary>Q1. 예전에 EC2 인스턴스를 분명 껐는데도 비용이 계속 나왔던 건 왜 그런 거야?</summary>

* **A1.** EC2 컴퓨터 전원을 껐더라도, 그 컴퓨터에 연결되어 있던 **가상 하드디스크(EBS), 사용하지 않고 방치된 고정 IP(Elastic IP), 혹은 켜져 있는 로드 밸런서(ALB)** 같은 주변 유료 자원들을 함께 삭제하지 않았기 때문입니다. 실습이 끝나면 연동된 리소스를 모두 종료해야 비용 폭탄을 막을 수 있습니다.

</details>

<details>
<summary>Q2. 이 프로젝트에서 필요한 전체적인 개념이랑 다른 팀원들 역할이 정확히 뭐야?</summary>

* **A2.** 인터넷상에 격리된 가상 네트워크 공간(VPC)을 만들고, 가상 컴퓨터(EC2)들을 배치한 뒤, 앱을 띄우고, 외부 공격을 막으며 감시하는 프로젝트입니다.
  * **강세영:** 컴퓨터가 배치될 도로(네트워크)를 깔고 가상 컴퓨터(EC2) 대수를 늘리는 역할 (Compute + Network)
  * **김태우:** 이 인프라 구축 과정을 마우스 클릭 대신 코드로 자동화하는 설계도를 짜는 역할 (IaC / Terraform)
    * **Q' : 여기서 말하는 인프라 구축이 EC2 인스턴스들에다가 똑같은 프로그램 설치하고, 세영이가 만드는 프로그램을 똑같이 깔고 그런 걸 말하는 건가? 만약에 EC2가 아니라 이미 프로그램이 깔려져 있는 사용자가 있다고 하면, 그 사람한테는 뭘 해주는 거야?**
    * **A' :** 아닙니다. 테라폼이 구축하는 '인프라'는 가상 컴퓨터 내부의 소프트웨어 설치가 아니라, 가상 컴퓨터(EC2) 자체를 새로 생성하고 가상 랜선을 연결하며 방화벽을 세우는 '외형 환경 규격' 자체를 코드로 만드는 것입니다. 컴퓨터 내부에 프로그램을 설치하고 세팅하는 것은 보통 Docker가 처리합니다. 만약 프로그램이 이미 깔려 있는 가상 컴퓨터가 존재한다면, 테라폼은 그 컴퓨터를 새로 만드는 대신 그 컴퓨터가 외부 인터넷과 통신할 수 있도록 라우터를 연결해 주거나 주변 네트워크 환경을 코드로 구성해 줍니다.
  * **박찬혁 (본인):** 외부 침입을 막는 방화벽(보안 그룹)을 설정하고, 비용 유출을 단속하며, 장애 대응 매뉴얼을 만드는 역할 (Security + SRE 보조)
    * **Q' : 외부 침입을 막는 로직이 뭔지 궁금해. 내가 알기로는 방화벽이라는 게 그냥 예전에 피해를 줬던, 아니면 앞으로 피해를 줄 가능성이 있는 애들을 사전에 차단하는 걸로 알고 있어. port#이랑 IP address 로만 하는 건가?**
    * **A' :** 찬혁님이 주로 담당하실 보안 그룹(Security Group)과 NACL 레벨에서는 정확히 Port 번호와 IP 주소만을 기반으로 화이트리스트 접근을 통제(L3/L4 방화벽 역할)합니다. 말씀하신 "악성 행위를 할 가능성이 있는 패킷을 사전에 분석해서 차단하는 로직"은 더 상위 계층인 애플리케이션 레벨(L7)의 방화벽인 AWS WAF(Web Application Firewall)나 IDS/IPS 같은 별도의 보안 솔루션이 처리합니다. 이번 프로젝트에서 찬혁님의 주 임무는 꼭 필요한 포트와 신뢰할 수 있는 IP 대역만 열어두고 나머지를 모두 차단하는 기초 방화벽 경계선을 치는 것입니다.
  * **권태욱:** 트래픽과 서버 상태를 24시간 감시하는 시스템(CloudWatch)을 구축하는 역할 (Observability)
    * **Q' : CloudWatch가 뭐야? 프로그램 이름이야?**
    * **A' :** 네, AWS가 자체적으로 제공하는 모니터링 및 관리 서비스(솔루션 프로그램)의 이름입니다. 가상 컴퓨터(EC2)에 부하가 얼마나 걸리는지, 네트워크 트래픽은 얼마나 들어오는지 등의 상태 지표를 수집하여 모니터 차트에 실시간 그래프로 그려주고, 문제가 생기면 알람을 발생시키는 종합 관제탑 소프트웨어입니다.

</details>

<details>
<summary>Q3. VPC랑 내가 학교에서 배운 네트워크 전공 지식이랑 어떻게 연결되는 거야?</summary>

* **A3.** 전공 수업에서 배운 사설 IP/공인 IP 개념, 서브넷 마스크, NAT 주소 변환 시스템이 AWS VPC에 소프트웨어로 그대로 구현되어 있습니다. VPC 내부 컴퓨터에는 **사설 IP**를 주고, 외부 인터넷과 통신할 때 **NAT 게이트웨이나 인터넷 게이트웨이**를 통해 **공인 IP**로 변환하여 통신합니다.

</details>

<details>
<summary>Q4. 외부 클라이언트(휴대폰)가 VPC 내부에 사설 IP를 쓰고 있는 EC2에는 어떻게 접속하는 거지?</summary>

* **A4.** 휴대폰이 사설 IP로 직접 찌르고 들어오는 것이 아닙니다. 외부에 공개된 공인 IP를 가진 **ALB(로드 밸런서)**가 휴대폰의 요청을 안전하게 대리 수신한 뒤, VPC 내부 네트워크를 통해 EC2의 사설 IP로 패킷을 배달해 주기 때문에 아무런 상관없이 정상 접속됩니다.

</details>

<details>
<summary>Q5. 인터넷 게이트웨이랑 ALB의 결정적인 차이가 뭐고, 내가 설정할 방화벽은 어디에 쳐지는 거야?</summary>

* **A5.** * **인터넷 게이트웨이:** L3(Network Layer)에서 IP 주소를 보고 우리 VPC 대역이 맞으면 통과시키는 대문 관문입니다.
  * **ALB:** L7(Application Layer)에서 HTTP 요청 내용(URL 경로 등)을 직접 뜯어보고 적절한 서버로 분산해 주는 안내소입니다.
  * **방화벽(보안 그룹):** EC2 가상 컴퓨터 바로 앞에서 특정 IP나 포트(웹용 80번, 관리자용 22번 등) 외의 악성 접근을 차단하는 차단막입니다.

</details>

<details>
<summary>Q6. 마크다운(.md) 파일이 정확히 뭐고 나도 토글 기능 쓸 수 있나?</summary>

* **A6.** 프로그래머들이 문서화할 때 쓰는 가벼운 텍스트 규격입니다. 노션에서 입력하는 방식(`#` 제목, `**` 굵게 등)과 똑같으며, 깃허브에 올리면 노션처럼 예쁜 웹페이지로 바뀝니다. 순수 문법에 토글 기호는 없지만, `<details>`와 `<summary>`라는 HTML 태그를 섞어 쓰면 접고 펼치는 토글을 구현할 수 있습니다.

</details>

<details>
<summary>Q7. 매일 git pull이랑 push를 해야 하는 흐름이 왜 중요하고, 만약 올리는 타이밍이 겹치면 어떻게 돼?</summary>

* **A7.** 여러 명이 협업하므로 **작업 전 git pull로 남이 올린 최신 코드를 받고 ➡️ 내 작업을 한 뒤 ➡️ git push로 올리는 것**이 기본 루틴입니다. 내가 pull을 받는 순간에 다른 사람이 push를 하더라도 Git 시스템이 알아서 차례대로 처리하므로 데이터가 깨지지 않습니다. 만약 동시 push로 거절당하더라도 **"다시 pull 받고 ➡️ push"** 하면 안전하게 합쳐집니다.

</details>

<details>
<summary>Q8. AWS 클라우드 안에 VPC가 있는 거잖아. 외부에서 AWS로 들어올 때만 공인 IP를 사용하는 줄 알았는데 각 VPC에도 공인 IP가 따로 있는 거야?</summary>

* **A8.** 아닙니다. VPC 자체는 `10.0.0.0/16` 같은 가상 사설 IP 대역만 가질 뿐, 자체적인 공인 IP를 기본 소유하고 있지 않습니다. 외부 인터넷 세상과 대화하기 위해 VPC의 출입구 역할을 하는 가상 장치(인터넷 게이트웨이, ALB)에 AWS가 소유한 거대한 공인 IP 자원 풀(Pool)에서 주소를 실시간으로 빌려와 연결(Mapping)하는 구조입니다. 외부에서는 이 공인 IP를 보고 찾아오며, 관문을 통과하는 순간 내부 사설 IP로 주소가 바뀌어 배달됩니다.

</details>

<details>
<summary>Q9. VPC 안에서 굳이 public 서브넷이랑 private 서브넷을 왜 번거롭게 나눠놓은 걸까?</summary>

* **A9.** 해커가 직접 찌르고 들어올 수 있는 외부 공격 통로(공인 IP 노출)를 최소한으로 줄이기 위한 보안의 핵심 설계 원칙 때문입니다. 외부 클라이언트를 직접 대면해야 하는 로드 밸런서(ALB) 등은 Public 서브넷에 오픈해 두고, 진짜 소중한 소스 코드가 도는 웹 서버(EC2)나 데이터베이스(DB)는 공인 IP가 없는 Private 서브넷 구역에 꽁꽁 숨겨서 인터넷망을 통한 해커의 직접적인 접근을 원천 차단합니다. (참고로 VPC와 서브넷을 쪼개는 설정 자체는 비용이 들지 않는 무료입니다.)

</details>

<details>
<summary>Q10. VPC 내부에서 지인들끼리 대화할 때(사설 IP 사용)는 집 공유기 안에서 기기들끼리 소통할 때처럼 네트워크 계층(L3)까지 안 보내고 링크 계층(L2)에서 처리하는 느낌인 건가?</summary>

* **A10.** 반은 맞고 반은 다릅니다! 완벽하게 같은 구조의 사설 대역(동일 서브넷 내부)끼리 통신할 때는 찬혁님 생각대로 L3 IP 주소는 목적지 확인용으로만 쓰이고 MAC 주소를 기반으로 L2(링크 계층) 가상 스위칭을 통해 다이렉트로 전달됩니다. 하지만 대역폭 규칙이 서로 다른 'Public 서브넷'에서 'Private 서브넷'으로 넘어갈 때는 아무리 같은 VPC(집안) 내부라 하더라도 사설 IP 주소를 보고 판단하는 L3(네트워크 계층) 가상 라우터의 도움을 받아 라우팅되어 이동합니다.

</details>

<details>
<summary>Q11. 근데 원래 public subnet에 있는 EC2는 공인 IP를 유동 IP로 줬던 거 같은데? 껐다 켜도 안 바뀌게 쭉 쓸 수 있게 해주는 탄력적 IP(EIP)는 뭐가 다르고 비용 단속할 때 뭘 조심해야 해?</summary>

* **A11.** 기본 공인 IP는 유동 IP가 맞습니다. 인스턴스를 종료(Stop)했다가 다시 시작(Start)하면 AWS가 주소를 회수해가고 새 주머니에서 다른 IP를 꺼내 줍니다. 반면 탄력적 IP(EIP)는 삭제하기 전까지 고정되는 정적 공인 IP라 웹 서비스 운영에 필수적입니다. 다만, SRE 비용 단속 시 중요한 포인트는 **"EIP를 만들어두고 EC2 컴퓨터를 꺼두거나 방치하면 오히려 벌금성 패널티 요금이 부과된다"**는 점입니다. 유한한 공인 IP 자원의 낭비를 막기 위한 AWS의 정책이므로, 실습이 끝나면 반드시 릴리스(Release, 삭제)해야 비용 유출을 막을 수 있습니다.

</details>

<details>
<summary>Q12. Private subnet에 있는 인스턴스가 외부 네트워크와 통신하려고 할 때 왜 굳이 public subnet에 있는 NAT 게이트웨이로 가? 그냥 바로 VPC의 인터넷 게이트웨이로 가면 되는 거 아니야?</summary>

* **A12.** 인터넷 게이트웨이(VPC 대문)는 외부 인터넷 세상과 연결되어 있어서 '공인 IP'가 적힌 패킷만 통과시켜 주기 때문입니다. 공인 IP가 아예 없는 Private EC2가 이 대문으로 바로 가면 문전박대(Drop)를 당합니다. 따라서 Public 서브넷에 상주하며 자체 공인 IP를 가진 'NAT 게이트웨이'를 대리인으로 삼아, 출발지 주소를 공인 IP로 싹 바꿔치기(NAT)한 뒤 대문을 통과해야 합니다. 이렇게 하면 안에서 밖으로 나가는 일방통행 문이 완성되어 외부 해커가 거꾸로 치고 들어오는 것을 막을 수 있습니다.

</details>

<details>
<summary>Q13. NAT 테이블은 뭐야? 언제 쓰는 거지? 그리고 NAT gateway 주소는 공인 IP야?</summary>

* **A13.** **NAT 게이트웨이**는 주소를 실제로 변환해 주는 '실물 가상 장치(기계)'이며, 당연히 외부 인터넷과 통신하기 위한 **100% 공인 IP(정확히는 탄력적 IP)**를 가지고 있습니다. 반면 **NAT 테이블**은 그 장치 내부에서 들고 있는 '주소 변환 기록 장부(메모)'입니다. Private EC2가 보낸 패킷의 사설 IP 주소를 공인 IP로 바꿔 보낼 때 장부에 기록해 두었다가, 나중에 외부 서버에서 답장이 돌아왔을 때 장부를 확인하고 원래 요청했던 사설 컴퓨터 주소로 정확하게 돌려주기 위해 사용합니다.

</details>

<details>
<summary>Q14. 만약 NAT 테이블 장부에 30개가 있는데 이미 다 쓰고 있다면? 결국 다른 public subnet에 있는 instance의 공인 IP를 빌려 쓴다는 거 아니야? 포트 개수가 6만 5천 개라 안 터진다고 해도 만약 인스턴스 개수가 그걸 넘어서면 어떡해?</summary>

* **A14.** **절대로 다른 Public EC2의 IP를 빌려 쓰지 않습니다.** NAT 게이트웨이는 전용 공인 IP 딱 1개만 사용하되, L4 계층의 포트 번호(Port Number, 0~65,535번)를 조합하여 장부 칸을 무지막지하게 쪼개 쓰는 **NAPT(Network Address Port Translation)** 기술을 씁니다. 통신이 오고 가는 순간에만 포트를 할당했다가 끝나는 즉시 회수하므로 1개 IP로도 수만 개의 연결을 처리합니다. 만약 대규모 트래픽으로 65,000개 포트가 진짜 꽉 차는 'NAT 포트 고갈' 현상이 터지면 패킷이 드롭되는데, 이때 SRE 담당자는 **1) NAT 게이트웨이에 공인 IP(EIP)를 추가로 더 사서 꽂아주거나(최대 8개, 약 51만 칸 확보), 2) 가용 영역별 다중화, 3) AWS 내부 통신망인 VPC 엔드포인트로 패킷을 우회시켜** 문제를 해결합니다.

## 3. 내가 이해한 내용

* **VPC의 본질과 계층 구조:** * AWS 클라우드 내에 우리만의 독립된 가상 사설망 방(VPC)을 먼저 파고, 그 안을 더 잘게 쪼개서 구역(서브넷)들을 나눈다.
  * VPC 자체는 철저히 사설 IP 대역만 사용하기 때문에, Public 서브넷의 EC2 내부에서 확인해도 사설 IP만 보이며 실제 외부 통신용 공인 IP는 대문(인터넷 게이트웨이)에 맵핑된다.
* **로컬 네트워크 통신의 원리:** * 같은 VPC(집안) 내부에서 기기끼리 통신할 때는 외부 인터넷망을 거치지 않는다. 단, 동일 서브넷 내부 통신은 L2(링크 계층) MAC 주소 기반으로 흐르지만, Public에서 Private 서브넷으로 넘어갈 때는 L3(네트워크 계층) 사설 라우터의 라우팅을 거쳐서 안전하게 이동한다.

## 5. 새로 알게 된 점

* **Public / Private 분리의 보안성:** 외부 노출 장치(ALB)만 Public에 두고 핵심 서버(EC2)는 공인 IP가 없는 Private에 숨김으로써 해커의 직접적인攻擊 표면을 원천 차단하는 설계 규칙을 깨달았다.
* **NAT 게이트웨이와 고정 IP(EIP)의 특성:** NAT 게이트웨이는 옆 동네 Public EC2의 IP를 빌려 쓰는 게 아니라, 독립적으로 발급받은 자기만의 전용 고정 공인 IP(EIP)를 가진다.
* **포트 번호 조합(NAPT)의 마법:** NAT 게이트웨이가 고작 1개의 공인 IP로 수만 개의 Private 연결을 소화할 수 있는 비결은 L4 레이어의 포트 번호(0~65,535)를 번호표처럼 임의로 다르게 붙여 장부(NAT 테이블)에 기록해 두는 기술 덕분임을 알았다.
* **SRE 관점의 비용 및 트래픽 관리:** VPC와 서브넷 설정 자체는 무료이지만 하드웨어 자원(EC2, ALB, NAT GW)에서 비용이 발생한다는 점, 그리고 탄력적 IP(EIP)를 연결하지 않고 방치하면 벌금성 패널티 요금이 나온다는 SRE 단속 포인트를 배웠다. 
  * 진짜로 65,000개 포트가 다 차는 '포트 고갈' 상황이 오면 남의 걸 뺏는 게 아니라 통신이 터지며, 이때는 공인 IP를 추가로 사서 꽂아 수용량을 수평 확장해야 한다.

## 6. Must / Optional 후보

### Must 후보
* **[Security]** 인프라 구축 후 EC2 바로 앞 구역에 꼭 필요한 최소한의 포트(22번, 80번 등)와 신뢰할 수 있는 IP 대역만 화이트리스트로 열어두는 기초 방화벽(보안 그룹) 세팅하기
* **[SRE 보조]** 매일 실습 및 프로젝트 작업이 끝나면 비용 유출을 막기 위해 켜져 있는 EC2 인스턴스 종료 단속하고, 사용하지 않는 탄력적 IP(EIP) 및 ALB, NAT 게이트웨이 리소스 즉시 삭제/릴리스 루틴 확립하기

### Optional 후보
* **[Architecture]** 대규모 트래픽 발생 시 NAT 포트 고갈을 원천 방어하기 위해, AWS 내부 서비스(S3 등) 통신 패킷은 외부 인터넷망을 타지 않고 내부 비공개 고속도로로 우회시키는 'VPC 엔드포인트' 연동 구조 검토하기
